# Serverless

`Serverless`（无服务器架构）是一个容易让人产生误解的名字。

**没有服务器，代码跑在哪里？**

事实上，**服务器依然存在，只是你不再需要关心它了。** 

为了可以由浅入深地理解，我们可以把 `Serverless` 的演进看作是 **去餐馆吃饭** 模式的进化。

---

## 1.什么是 Serverless？

在传统模式下，如果你想开一家餐馆（运行一个程序），你需要买地、盖房子、买厨具、招厨师（购买服务器、装系统、配置环境、维护带宽）。

即便没有客人的时候，你也要付房租和工资。

`Serverless` 则是 **外卖平台** 模式:

- **无需管理：** 你只需要提供菜谱（代码），平台负责所有的后勤。
- **按需付费：** 只有人下单（触发请求）时，你才付钱。没人下单，你的成本为零。
- **自动扩容：** 如果突然有一万个人下单，平台会自动调集一万个厨房同时开火，你不需要担心厨房坐不下。

### 核心组成：`FaaS` + `BaaS`

`Serverless` 通常由这两部分组成：

1. `FaaS`（`Function as a Service`，函数即服务）：你的业务逻辑。比如**计算折扣**的这一段小代码。常见产品：`AWS Lambda`，阿里云 `Function Compute`（`FC`）。
2. `BaaS`（`Backend as a Service`，后端即服务）：现成的功能模块。比如数据库、用户登录、文件存储。你不需要自己写，直接调用 `API`。常见产品：`Firebase`，阿里云 `OSS`。

---

## 2.它为什么自动？

当你上传一段代码到 `Serverless` 平台后，它是如何工作的？

1. **事件驱动**（`Event-Driven`）：代码平时是**死的**。只有当某个事件发生（比如用户访问网页、文件上传成功、定时器到点）时，平台才会**唤醒**代码。
2. **冷启动**（`Cold Start`）：
   - 当第一个请求进来，平台会瞬间创建一个轻量级容器（如 `Docker` 或 `Firecracker` 虚拟机），把你的代码塞进去并运行。这个过程叫**冷启动**，可能会有几十毫秒到几秒的延迟。
   - 运行完后，容器会保留一小会（**热启动**），如果一直没请求，容器就被销毁。
3. **状态无关**（`Stateless`）：
   - 函数运行完就消失了。这意味着你不能把数据存在函数的内存里，必须存在外部的数据库（`BaaS`）中。

---

## 3.优缺点

`Serverless` 并非万能药，它有非常明显的性格特征：

### 为什么用它？（优点）

- **极低成本：** 适合流量波动巨大的应用。比如抢购活动，平时没流量不花钱，高峰期自动顶上去。
- **专注业务：** 开发者不再是**运维工程师**，只要代码逻辑对，程序就能跑。
- **上线快：** 几秒钟就能部署全球。

### 它的软肋在哪里？（缺点）

- **冷启动延迟：** 对于要求毫秒级响应的实时交易系统，冷启动可能是个麻烦。
- **调试困难：** 因为环境在云端，本地很难 100% 模拟生产环境的报错。
- **供应商锁定：** 如果你深度使用了阿里云的 `Serverless` 接口，想迁移到 `AWS` 会非常痛苦。
- **不适合长任务：** 如果一个计算任务要跑 1 小时，`Serverless` 的按时计费反而比包月服务器更贵。

---

## 4.场景对比

| 场景类型 | 是否推荐 `Serverless` | 原因 |
| :--- | :--- | :--- |
| **小程序/移动端后端** | ⭐⭐⭐⭐⭐ | 大多是碎片化请求，非常省钱。 |
| **图片/视频处理** | ⭐⭐⭐⭐⭐ | 上传一张图触发一次压缩，完美匹配。 |
| **定时任务**（`Cron`） | ⭐⭐⭐⭐⭐ | 每天跑一次，没必要为此买台 24 小时开机的服务器。 |
| **大型 `ERP`/复杂单体架构** | ⭐ | 逻辑太重，组件依赖多，拆分 `FaaS` 的成本极高。 |
| **高性能数据库/高频交易** | ⭐⭐ | 状态管理复杂，对延迟极度敏感。 |

## 5.实战案例：带自动水印的图片处理工具

下面我们直接动手实现一个最经典的 `Serverless` **实战案例**：**一个带自动水印功能的图片处理工具**。

这个案例能让你瞬间理解 `Serverless` 的**事件驱动**本质：**用户上传图片（事件） -> 触发函数（运行） -> 生成水印图（结果）**。

---

### 技术选型

- `FaaS`（函数计算）：比如阿里云 `FC` 或 `AWS Lambda`。
- `BaaS`（对象存储）：比如阿里云 `OSS` 或 `AWS S3`。

---

### 核心代码实现

这段代码逻辑很简单：读取上传的图片，用 `sharp` 库打上文字水印，然后存回去。

```js
const OSS = require('ali-oss');
const sharp = require('sharp'); // 强大的图片处理库

exports.handler = async (event, context, callback) => {
    // 1. 获取触发事件的信息（谁上传了哪个文件）
    const eventObj = JSON.parse(event.toString());
    const objectName = eventObj.events[0].oss.object.key;
    const bucketName = eventObj.events[0].oss.bucket.name;

    // 2. 配置 OSS 客户端
    const client = new OSS({
        region: 'oss-cn-hangzhou',
        accessKeyId: process.env.AK,
        accessKeySecret: process.env.SK,
        bucket: bucketName
    });

    try {
        // 3. 下载原图到内存
        const { content } = await client.get(objectName);

        // 4. 使用 sharp 处理图片：添加水印
        const watermarkedImage = await sharp(content)
            .composite([{
                input: Buffer.from('<svg><text x="10" y="30" font-size="30" fill="#fff">My Watermark</text></svg>'),
                gravity: 'southeast'
            }])
            .toBuffer();

        // 5. 将处理后的图片存入 'output/' 目录
        await client.put(`output/${objectName}`, watermarkedImage);

        console.log('水印添加成功！');
        callback(null, 'Success');
    } catch (err) {
        callback(err);
    }
};
```

---

### 部署与运行逻辑

1. **上传代码：** 你把这段代码打包上传到云平台的**函数计算**控制台。
2. **设置触发器：** 配置一个 `OSS` **触发器**。
   - *条件：* 当 `images/` 目录下有新文件创建时。
   - *动作：* 立即执行上面的函数。
3. **见证奇迹：**
   - 你本地上传 `photo.jpg` 到云端的 `images/` 目录。
   - **这一秒：** 云平台检测到文件，自动分配一台**临时小电脑**，装好 `Node.js` 环境，运行你的代码。
   - **下一秒：** 处理完成，图片出现在 `output/` 目录。
   - **再下一秒：** **临时小电脑**被销毁，云平台停止计费。

---

### 为什么这个案例最能体现 `Serverless` 的优势？

- **零运维：** 你不需要写一个 24 小时运行的后端服务去监听文件夹，云平台帮你盯着。
- **极致弹性：** 如果你一次性上传 1000 张图，云平台会瞬间开启 1000 个函数并行处理，几秒钟全部搞定；如果是传统的 1 核服务器，可能要排队跑半小时。
- **省钱：** 如果一个月没人上传图片，你一分钱都不用付。

## 6.Serverless 应用场景

对于前端开发来说，`Serverless` 的兴起本质上是 **前端能力的边界扩张**。

它让前端工程师无需掌握复杂的服务器运维（如 `Nginx` 配置、`Docker` 镜像、数据库集群），就能实现**全栈**功能。

以下是目前前端领域最火的 **4 个** `Serverless` **应用场景**及其具体例子：

---

### 1. `BFF`

这是目前大厂前端团队应用最广的场景。

- **痛点：** 后端提供的接口往往是基于**实体**的（比如用户接口、订单接口），但前端页面需要的是**聚合**数据。如果直接在手机端发 5 个请求，弱网下会非常卡。
- `Serverless` **方案：** 前端自己写一个 `Serverless` 函数作为中转站。
- **例子：**
  - **场景：** 电商 `App` 首页。
  - **逻辑：** 页面请求一个 `/api/home`。这个 `Serverless` 函数在云端**同时**调用**用户信息**、**推荐商品**、**天气插件**、**促销横幅** 4 个微服务接口，然后把数据剪裁成页面刚好需要的格式，一次性返回。
  - **价值：** 减少网络开销，且接口逻辑由最懂页面的前端控制，不需要求后端改接口。

---

### 2. `SSR` / `ISR`

随着 `Vue 3` 和 `React` 的普及，单页应用（`SPA`）的 `SEO` 和首屏加载成了难题。

- **痛点：** 传统的 `SSR` 需要维护一个 `Node.js` 服务器（如 `PM2` 管理），一旦流量暴增，服务器很容易宕机。
- `Serverless` **方案：** 利用 `Serverless` 的**自动扩容**特性，将每一个路由请求交给一个函数去处理渲染。
- **例子：**
  - `Vercel` / `Netlify`：当你部署一个 `Nuxt.js` 或 `Next.js` 项目时，它们底层会自动将每个 `API` 和页面路由转换成 `Serverless` 函数。
  - `ISR`（增量静态再生）：像新闻或博客网站，平时是静态页面，但当数据更新时，`Serverless` 函数会在后台重新生成静态 `HTML` 并推送给 `CDN`，用户访问永远是毫秒级。

---

### 3. 自动化任务与钩子

处理那些**异步**、**重计算**或**定时**的任务。

- **痛点：** 前端上传一张 10MB 的高清头像，如果在浏览器里切图（`Canvas`），低端手机会直接卡死。
- `Serverless` **方案：** 触发式处理。
- **例子：**
  - **头像缩略图：** 用户上传原图到 `OSS`（对象存储），自动触发一个 `Serverless` 函数，在云端调用 `Sharp` 库生成 100x100 的缩略图，并存回云端。
  - `CI/CD` 机器人：当你在 `GitHub` 提交代码（`Push` 事件），触发 `Serverless` 函数，自动给你的飞书/钉钉群发一个**某某代码已提交，正在构建**的通知。

---

### 4. 边缘计算

这是目前 `Serverless` 最前沿的方向，代码不跑在中心服务器，而是跑在离用户最近的 `CDN` 节点。

- **痛点：** 哪怕 `Serverless` 很快，如果服务器在美国，中国用户访问依然有延迟。
- `Serverless` **方案：** `Edge Functions`（如 `Cloudflare Workers`）。
- **例子：**
  - `A/B` 测试：在 `CDN` 边缘节点直接拦截请求。函数判断用户的 `Cookie`，如果是 `A` 组就重定向到 `index_a.html`，如果是 `B` 组就去 `index_b.html`。
  - **个性化广告/权限拦截：** 在请求到达服务器之前，在**边缘**就把没有登录的用户拦截掉，或者根据用户 `IP` 地理位置直接修改返回的内容（比如北京用户看到北京的天气）。

---

### 总结：

如果你想在项目中体现 `Serverless` 能力，可以尝试以下组合：

| 维度 | 推荐工具 | 适合练习的例子 |
| :--- | :--- | :--- |
| **快速起步** | `Vercel` / `Netlify` | 部署一个 `Vue 3` + `Vite` 的全栈个人博客。 |
| **国内大厂方案** | 阿里云 `FC` / 腾讯云 `SCF` | 写一个自动爬取某技术社区文章并推送到微信的脚本。 |
| **极致性能** | `Cloudflare Workers` | 实现一个根据地理位置重定向页面的 `Edge Middleware`。 |
| **本地开发** | `Serverless Devs`（`s.yml`） | 学习如何用代码定义基础设施（`IaC`）。 |

## 7.常见问题

说到 `Serverless`，有两个核心痛点——**冷启动** 和 **状态管理** 是必须要了解的。

---

### 1. 如何优化**冷启动**延迟？


> `Serverless` 虽好，但函数冷启动会导致首次访问变慢，前端怎么解决？

**答案需要从多维度切入：**

- **架构层：** 预留实例。对于高频核心接口，可以在云平台配置 **预留实例**，让函数始终处于 **热态**，像传统服务器一样秒回。
- **代码层：精简包体积。**
  - **按需加载：** 不要在 `Node.js` 函数里 `require` 整个大型库（比如 `lodash` 或 `aws-sdk`），只引入需要的子模块。
  - **打包优化：** 使用 `Vite` 或 `esbuild` 将云函数代码压缩混淆，减少平台加载代码包的 `I/O` 时间。
- **平台层：选择轻量运行时。** `Node.js` 的冷启动通常比 `Java` 或 `Python` 快。如果追求极致，可以尝试 `Edge Runtime`（如 `Cloudflare Workers`），它基于 `V8 Isolate` 技术，冷启动几乎为 0。
- **用户体验层：** 利用前端的 `Loading` 态、骨架屏，或者在用户鼠标 `Hover` 到按钮时就提前预触发函数请求（`Pre-fetch`），抵消掉冷启动的时间差。

---

### 2.`Serverless` 是无状态的，如何处理登录态？

> `Serverless` 函数执行完就销毁了，没有 `Session`，怎么处理用户登录？

**策略分两种：**

- **核心方案：** `JWT`。放弃传统的 `Session` 模式，改用 `JWT`。用户信息加密在 `Token` 中，前端每次请求带上 `Token`，`Serverless` 函数只需要一套解密算法（和环境变量里的 `Secret`）就能验证身份，不需要查询本地内存。
- **外部状态机：** `Redis`。如果业务逻辑一定要存状态（比如验证码、购物车），必须借助 `BaaS`。使用云平台提供的分布式缓存 `Redis`，函数启动后连接 `Redis` 读写数据，执行完断开。

---

## 3.`Serverless` 的成本真的比 `ECS` 低吗？


> 什么情况下你会建议公司从传统服务器迁移到 `Serverless`？

**这里权衡考虑：**

- **流量特征：** 如果流量是**脉冲式**的（比如抢购、准点推送）或者**长尾式**的（比如内部管理后台，半天没人点一下），`Serverless` 极省。
- **运维成本：** `Serverless` 省去了修补系统漏洞、扩容、打补丁的人力成本。对于小团队，**人力的钱** 远比 **机器的钱** 贵。
- **反例：** 如果流量非常平稳且巨大（比如 24 小时高并发），传统包月服务器反而更便宜。
