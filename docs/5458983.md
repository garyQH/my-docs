## await nextTick() 的使用场景

> 简单来说，加不加 `await nextTick()` 的区别在于：你是想在 **数据变了** 之后立即做事，还是想在 **页面渲染完了** 之后再做事。

在 `Vue` 中，**`DOM` 的更新是异步的**。

---

### 1. 不加 `await nextTick()`

```js
const count = ref(0)
```

当你执行 `count.value++` 时，`Vue` 会立刻察觉到数据变化，但它 **不会** 立即去更新 `DOM`。
它会把这个更新任务丢进一个队列里，等待同步代码全部执行完后，再统一进行渲染。

- **执行结果：** 数据已经变成了新值，但如果你此时去获取 `DOM` 元素（比如 `document.getElementById('counter').innerText`），你拿到的依然是**旧的**内容。
- **适用场景：** 你只需要改变逻辑数据，不关心 `UI` 什么时候刷新。

### 2. 加上 `await nextTick()`

`nextTick` 是一个 `Promise`，它会等待当前的 **DOM 更新队列** 被清空（即页面已经根据最新的数据重新渲染完毕）后才会被解析（`resolve`）。

- **执行结果：** 当代码运行到 `nextTick` 之后的行时，`Vue` 已经完成了对 `DOM` 的修改。此时你去获取 `DOM` 元素，拿到的是**新的**内容。
- **适用场景：** 你需要在数据改变后，立刻操作受该数据影响的 `DOM`。比如：
  - 改变列表数据后，立即获取列表的高度。
  - 显示一个输入框后，立即调用 `.focus()` 让它获取焦点。
  - 集成第三方非 `Vue` 的插件（如 `ECharts` 或 `jQuery` 插件），需要确保容器已经更新。

---

### 形象对比

我们可以把 `Vue` 的更新机制比作 **餐厅出菜**：

| 步骤 | 行为 | 对应代码 |
| :--- | :--- | :--- |
| **1. 点菜** | 你告诉服务员（`Vue`）你要加个菜（`count++`）。 | `count.value++` |
| **2. 等待** | 服务员不会跑一趟厨房就上一道菜，他会等桌上其他菜点完，一起去厨房（异步队列）。 | (同步代码继续执行) |
| **3. 出菜** | 厨师做好了，菜端上了桌（`DOM` 更新完毕）。 | **`nextTick` 结束** |

---

### 代码演示区别

```js
async function increment() {
  count.value++

  // 情况 A: 不加 nextTick
  console.log('立刻获取 DOM:', document.getElementById('num').innerText) 
  // 结果：可能是旧值 (例如 0)

  await nextTick()

  // 情况 B: 加上 nextTick
  console.log('NextTick 后获取 DOM:', document.getElementById('num').innerText) 
  // 结果：一定是新值 (例如 1)
}
```

---

### 总结

- **不加：** 性能更高（减少了等待），适合纯数据逻辑。
- **加上：** 保证同步，适合需要"感知"页面变化的场景。
