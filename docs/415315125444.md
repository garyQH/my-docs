# 记一次`AI`项目

> `ChatBI`智能数据分析助手

项目背景：

为了降低非技术用户查询企业级数据资产的门槛，基于公司现有的**数据资产管理平台**，开发了一套基于 `LLM` 的对话式报表系统。

用户通过自然语言输入，系统自动生成对应的 `SQL` 并在前端实时渲染可视化报表。

其中遇到了一些问题，解决方案如下记录：

## 全链路流式交互方案实现

- 基于 **`SSE` 协议** 实现了前端流式数据解析。
- 针对大模型长文本输出导致的页面卡顿问题，设计了**双缓冲队列机制**，确保 `Markdown` 文本、`SQL` 代码块与 `ECharts` 配置项能平滑、有序地异步增量渲染。
- 利用自定义 `Markdown` 插件扩展了图表组件指令，实现了对话框内图文混排的极致体验。

### 1. 协议选择：为什么是 `SSE` 而不是 `WebSocket` 或普通 `HTTP`？

**`SSE` 的优势：**

- **流式传输：** `LLM` 是逐个 `token` 生成的。`SSE` 允许服务器在生成时就不断向前端推送，用户不需要等待几十秒生成完才能看到内容，体验从**加载中**变成了**打字机中**。
- **轻量化：** 相比 `WebSocket`，`SSE` 是基于标准 `HTTP` 协议的单向推送，天然支持断线重连，且对负载均衡更加友好（无需处理心跳包）。

> 因为 `SSE` 的原生支持并不友好，这里使用了微软开源的一个库 [fetch-event-source](https://github.com/Azure/fetch-event-source)，相关可以查看源文档。

### 2. 核心难点：双缓冲队列机制

**痛点：为什么会卡顿？**

- **高频重绘：** `LLM` 推送速度极快（尤其在内网环境下）。如果每收到一个字符就去触发 `Vue` 的响应式更新并让 `markdown-it` 重新渲染一遍全文，会导致 `CPU` 瞬间飙升，页面滚动卡死。
- **`DOM` 压力：** 频繁的 `DOM` 树对比和重排是前端性能的头号杀手。

**解决方案：双缓冲机制**

- **接收缓冲：** 前端 `ReadableStream` 持续接收原始字符串，存入一个快速增长的字符串变量中。
- **消费时钟：** 开启一个定时器（如 `requestAnimationFrame` 或 `50-100ms` 的 `setInterval`）。
- **批量渲染：** 只有当定时器触发时，才将缓冲区内的最新全量文本同步给 `Vue` 的 `ref` 变量。

### 3. 解析与扩展：自定义插件

`ChatBI` 最大的难点在于：它不只是文字，还有**图表**和 `SQL`。

解决方案也很简单：

- **定制插件：** 通过自定义解析器，解析代码标识符，使用自定义组件渲染。

对于图文混排的实现：

- **异步渲染：** 在流式传输过程中，图表配置可能还没传输完（`JSON` 不完整）。
- **状态处理：** 需要**状态组件**且具备**加载中**、**解析中**、**渲染完成**三种状态。在 `JSON` 闭合之前，通用组件可以显示一个 `Loading` 动画，一旦监听到闭合符，立即触发**渲染组件**初始化。

## 可插拔式图表协议渲染引擎

设计了一套 `JSON Schema` 驱动的图表渲染协议。

### 1. 核心思想：`JSON Schema` 协议设计

**痛点：** `LLM` 输出的文本是不可预测的。如果直接让它写 `js` 代码去初始化 `ECharts`，会存在安全风险（注入攻击）且极难维护。

**个人方案：**

定义了一套标准的 `JSON` 格式（`Schema`），告诉模型：**你只需要按照这个格式给我数据，剩下的我来办**。

- **协议示例：**

```js
{
  "component": "BarChart",
  "props": {
    "title": "2026年Q1销售额",
    "dataset": {
      "columns": ["月份", "金额"],
      "source": [["1月", 100], ["2月", 150]]
    },
    "stack": true
  },
  "actions": {
    "click": "drilldown"
  }
}
```

这套协议将**业务意图（做什么图）**与**渲染实现（怎么画图）**彻底解耦。

数据模型只需关注数据结构，而无需关心 `ECharts` 的繁琐配置项。

这里需要在系统提示词中植入一套**图表决策树**。

例如你是一个资深数据分析师。根据用户的问题，请从以下组件中选择最合适的一个：

- 如果涉及时间趋势，使用 `LineChart`（折线图）。
- 如果涉及占比分布，使用 `PieChart`（饼图）。
- 如果涉及数值对比，使用 `BarChart`（柱状图）。
- 如果只是明细查询，使用 `Table`（表格）。
- 必须输出如下 `JSON` 格式：`{ "component": "...", "data": [...] }`。

> 通过系统提示词（`System Prompt`）约束大模型的输出边界，这能让模型的决策准确率达到 `95%` 以上。

### 2. 实现机制：动态映射引擎

**技术点：** 动态组件（`<component :is="...">`）。

**具体逻辑：**

1. **解析层：** 前端接收到模型生成的 `JSON` 后，通过一个 `ChartFactory` 转换器进行校验。
2. **映射层：** 根据 `component` 字段，动态匹配前端预设的封装组件（如 `BarChart.vue`、`PieChart.vue`、`Sunburst.vue`）。
3. **注入层：** 将协议中的 `props` 动态绑定到组件上。

总结：构建了一个动态渲染引擎。

它像一个**翻译官**，将模型输出的逻辑配置实时映射为 `Vue` 动态组件。

这种**插件化**（`Pluggable`）的设计，使得以后增加新图表类型（如词云、桑基图）时，只需增加一个标准组件，而无需修改核心交互逻辑。

### 3. 高级交互：图表联动与多维下钻

其实静态图表不难，难在 `AI` 生成的图表还能**交互**。

- **联动：** 比如点击**饼图**的一个区域，侧边的**列表**自动过滤。
  - **实现：** 在协议中定义 `events` 字段，前端解析协议后，通过 `Event Bus` 或 `Pinia` 触发动作。
- **下钻：** 点击**年份**进入**月份**。
  - **实现：** 当点击事件发生时，前端自动发起一个新的 `AI` 请求，并在 `Prompt` 中加入当前点击的维度参数（如：**请展示 2026 年 2 月的具体明细**），从而实现**对话式的多维分析**。

### 4. 可插拔与解耦的价值

1. **稳定性：** 无论模型怎么**胡言乱语**，只要不符合 `Schema`，引擎就会报错并展示兜底 `UI`，不会导致整个页面白屏。
2. **易维护性：** 业务逻辑不再堆叠在对话列表里。如果 `ECharts` 升级了，你只需要改底层的 `BaseChart` 组件，上层 `AI` 逻辑完全不用动。
3. **扩展性：** 这套引擎不仅能渲染图表，还能渲染表格、进度条甚至自定义的业务看板。

### 总结

1. 通过一套`JSON Schema` 驱动的渲染引擎来解决自定义渲染问题。
2. 传统的做法是让 `AI` 直接输出 `ECharts` 配置，但这会导致前端逻辑极其臃肿且难以校验。我通过制定标准协议，将 `AI` 定位为**结构化数据提供者**，前端则负责**协议解析与组件映射**。
3. 通过这种架构，我不仅实现了图表的流式平滑渲染，还通过**事件委托机制**解决了 `AI` 图表的联动与下钻问题。

### `PS`

- 如果模型输出的 `JSON` 格式不完整（流式传输中），你的引擎会报错吗？
- 可以利用 `JSON` 状态机 或简单的 `try-catch` 预检查。在 `JSON` 完全闭合之前，我会展示一个骨架屏（`Skeleton`）或 `Loading` 态，只有当检测到合法的结束符时，才触发 `ECharts` 的初始化实例，确保用户体验连贯。

## 大模型 Context 协同与 Schema 注入

利用原系统沉淀的元数据 `SDK`，在前端实现了 `Context` 预处理层。

根据用户当前的资产搜索上下文，动态按需注入表结构描述，降低了大模型生成 `SQL` 的幻觉率，提升准确度。

> `LLM` 最怕的是**盲目猜测**。如果用户问**去年的销售额是多少？**，模型不知道你的数据库里是叫 `sales_order` 表还是 `t_order` 表，字段是 `amount` 还是 `total_price`。这就导致了**幻觉**（乱写 `SQL`）。

核心的解决方案，就是我们需要构建一个信息中转站，在问题发出前，给模型带上**参考资料**。

> 这一步也可以后端来做，后面会解释原因。

### 1. 核心技术：`Context` 预处理层

**痛点：**

模型虽然聪明，但它没权限实时看你的数据库字典。如果把几千张表的结构全部塞进 `Prompt`，会超过模型的长度限制，而且非常烧性能。

**我们的方案：**

在前端实现一个**按需注入**的逻辑。

- **动作：** 当用户在聊天框输入文字时，前端先不发给 `AI`，而是先过一遍**预处理层**。
- **逻辑：** 提取用户问题里的关键词（如**资产**、**质量**、**血缘**），利用之前业务**元数据管理系统**里封装好的 `SDK`，去本地缓存检索相关的表结构描述。

### 2. `Schema` 注入

**实现细节：**

我们将检索到的元数据转化为一段精简文本，拼接到 `System Prompt` 中。

- **注入前：** 请帮我写一个查询资产质量的 `SQL`。（模型：我猜表名是 `asset`）
- **注入后：**
  - **参考信息：** 表 `meta_asset_info`（字段：`id`、`name`、`quality_score`、`update_time`）。
  - **用户问题：** 请帮我写一个查询资产质量的 `SQL`。

> 由于有了参考，模型能准确写出 `SELECT quality_score FROM meta_asset_info`。

### 3. 做到了什么？

1. **减少了名称错误：** 解决了表名、字段名大小写或缩写不一致导致的 `SQL` 执行失败。
2. **约束了关联关系：** 告诉模型哪些表之间有外键（`Join Key`），避免模型乱连表。
3. **缩小了搜索空间：** 只给相关的 5 张表，而不是全库 500 张表，干扰项变少了。

### `PS`

#### 这种逻辑为什么放在前端做，而不是后端

- 这是一个**协同架构**的选择。
- 第一，**响应速度：** 前端维护了用户的**当前搜索上下文**（比如用户刚才正在看某张表的详情），我可以利用这些即时信息进行**零延迟**注入。
- 第二，**交互闭环：** 前端可以根据注入后的 `Context` 长度，动态提醒用户**当前引用了 3 张表的信息**，甚至让用户手动勾选**参考哪张表**，这种**透明的交互**是后端难以直接实现的。
- 第三，**解耦：** 后端只负责提供原子化的元数据 `API`，由前端根据 `UI` 场景决定**喂**给 `AI` 什么信息，这样逻辑更灵活。

#### 如果 `Schema` 太大，超出了 `Prompt` 限制怎么办？

- 需要进行`Token` 裁剪。只保留 `表名`、`字段名` 和 `字段备注`，剔除掉索引、触发器等对生成 `SQL` 无关的元信息。如果还是太大，需要引入 **`RAG` 向量搜索**，只注入相关度最高的前 3 个表结构。

## 复杂会话状态与性能优化

针对长对话场景下的 `DOM` 压力，采用**虚拟列表**（`Virtual List`）技术重构了消息流容器，保证在数百条对话记录下仍能维持丝滑滚动。

### 1. 虚拟列表：解决长对话的 `DOM` 压力

**痛点：**

对话式 `AI` 的特点是**无限增长**。随着用户交流深入，对话框内会堆积大量文本、`SQL` 代码块，甚至是复杂的 `ECharts` 图表。

- **`DOM` 溢出：** 如果有 500 条消息，每条消息包含复杂的 `Markdown` 和 `Canvas`（图表），`DOM` 节点数可能过万。
- **渲染掉帧：** 浏览器在滚动时需要不断计算样式和重排（`Reflow`），导致页面卡顿、掉帧，甚至内存溢出。

**解决方案：**

只渲染**可视区域**及其前后的少量 `DOM` 节点，其余部分用空白占位符（`Padding`）顶替。

**技术难点（高度自适应）：**

- 普通的虚拟列表要求每项高度固定，但 `AI` 对话框的高度是**动态且不可知**的（有的消息两行，有的带大图表）。

**实现细节：**

1. **动态测量：** 利用 `ResizeObserver` 监听每个消息组件的大小。
2. **位置索引缓存：** 维护一个偏移量地图（`Offset Map`），记录每条消息的 `top` 值和 `height`。
3. **滚动补偿：** 在滚动过程中动态计算当前应该显示哪几条消息，并计算滚动条的模拟高度。

### 2. 对话快照与状态持久化（没有实现）

**场景：** 用户在**血缘分析图**中发现某个数据指标异常，想直接点按钮问 `AI`：**这张表的质量得分为什么下降了？**

**实现逻辑：**

1. **快照定义：** 定义一个 `SessionContext` 协议，包含：`currentTableId`（当前表 `ID`）、`lineagePath`（血缘路径）、`filters`（当前的筛选条件）。
2. **状态挂载：** 利用 `Pinia` 或 `Redux` 维护一个全局的对话状态。当用户点击跳转 `AI`时，将血缘图的当前状态**快照**下来。
3. **路由带参：** 通过路由参数或状态库，将快照信息传递给 `ChatBI` 模块。
4. **自动触发：** `AI` 模块初始化时，识别到有**传入上下文**，会自动触发一个隐藏的预设 `Prompt`（例如：**我正在查看表 A，请分析它的血缘质量**），实现无缝衔接。

### 3. 滚出屏幕被销毁的图表，再滚回来怎么保证不闪烁？（没有实现）

- **答：** 可以做**状态持久化缓存**。当图表组件因为虚拟列表滚动被卸载（`Unmount`）前，我会将其配置项（`Option`）和当前的数据快照保存在内存中。当它重新进入可视区挂载时，直接从内存读取数据进行同步初始化，避免重新发起后端请求，从而实现视觉上的**无缝还原**。

### 4. 长对话不仅 `DOM` 压力大，内存压力也大，你怎么处理？（没有实现）

- **答：** 除了虚拟列表减少 `DOM` 节点，我还做了**数据分片存储**。对于极长的对话（如超过 1000 条），我会将早期的对话内容从当前的响应式状态（`Store`）中移出，序列化后存入浏览器的 `IndexedDB`。只有当用户向上滚动查阅历史时，才按需异步加载回来。这样可以保证 `Vue` 的响应式系统不会因为追踪海量数据而变慢。

### 4. 总结

> 针对性能，通过**自适应高度的虚拟列表**重构了消息容器，通过 `ResizeObserver` 和**偏移量预计算**，将数千个复杂节点的渲染压力降低到了常数级别，确保了 `60fps` 的滚动表现。
>
> 针对交互，设计了一套**跨模块的 `Context` 协议**，实现了从血缘图到 `AI` 助手的**一键带入**。通过快照机制和 `IndexedDB` 持久化，确保了用户在复杂分析链路中，`AI` 始终能精准掌握当前的业务上下文，避免了重复输入。

## 工程化资产复用（中台化）

> 将原资产平台的**数据字典**与**权限校验**模块抽象为独立业务层，使 `ChatBI` 能无缝接入原有的 `RBAC` 权限体系，确保 `AI` 检索结果不逾越数据安全边界。

### 1. 为什么中台化复用？

**痛点：**

如果 `ChatBI` 是一个独立项目，很容易变成**信息孤岛**。

需要重新对接登录、重新写权限判断、重新维护数据字典。

一旦所依托平台的权限逻辑改了，`ChatBI` 就会产生安全漏洞。

**解决方案：**

采用**私有 `NPM` 包**的形式，将核心逻辑抽离成**业务中台层**。

- **数据字典 `SDK`：** 包含字段翻译、枚举值映射、敏感字段脱敏规则。
- **权限校验拦截器：** 包含 `RBAC`（基于角色的权限访问控制）的判定逻辑。

### 2. 如何确保 `LLM` 不逾越安全边界？

实现分为三层过滤：

#### 第一层：`Schema` 注入过滤

在将表结构（`Schema`）喂给 `AI` 之前，前端 `SDK` 会先调用权限模块：

- **逻辑：** `PermissionSDK.getAccessibleTables(userId)`。
- **动作：** 只有用户在原资产平台有权限查看的表，才会被加入到 `Prompt` 的上下文（`Context`）中。
- **效果：** `LLM` 压根不知道那些敏感表的存在，从物理上隔绝了**越权提问**。

#### 第二层：动态 `UI` 权限受控（没有实现）

- **逻辑：** `AI` 生成的 `JSON` 协议中如果包含某张表，前端渲染引擎会再次校验。
- **动作：** 在 `Table` 或 `Chart` 组件挂载前，通过全局指令（如 `v-auth`）判断。如果无权访问，`UI` 会显示**无权查看该资产**，而不是直接展示数据。

#### 第三层：数据脱敏机制（没有实现）

- **逻辑：** 调取**数据字典**模块的脱敏规则（`Masking Rules`）。
- **动作：** 如果 `AI` 返回的数据包含手机号、身份证等，前端组件自动根据字典定义进行星号遮盖（如 `138****8888`）。

### 4. 总结

> 在开发 `ChatBI` 时，我没有将其视为一个孤立插件，而是基于**中台化思维**进行了工程架构设计。
>
> 将原平台的 `RBAC` 权限体系与数据字典模块抽离成了**通用的业务 `SDK`**。在 `AI` 交互链路中，实现了**双重安全校验**：
>
> 1. 在 **`Prompt` 构建阶段**，利用 `SDK` 动态过滤掉用户无权的元数据，从源头防止 `AI` 产生越权幻觉。
> 2. 在 **`UI` 渲染阶段**，复用主站的权限指令确保敏感数据的安全展示。
