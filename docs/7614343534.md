## 按下暂停键的 AbortController

在现代 `Web` 开发中，**`AbortController`** 是中止异步操作（尤其是网络请求）的标准方案。

它的核心逻辑就像是一个**遥控器**：你把信号（`Signal`）交给请求，一旦你按下开关（`Abort`），请求就会立即停止。

---

### 1. `AbortController` 的核心机制

在使用具体框架前，先理解它的基本流程：

1. **实例化**：`const controller = new AbortController();`
2. **获取信号**：`const signal = controller.signal;`
3. **触发中止**：`controller.abort();`

---

### 2. `Fetch API` 中的用法

`Fetch` 是原生支持 `AbortController` 的。你只需要将 `signal` 放入配置对象中。

```js
const controller = new AbortController();

fetch('https://api.example.com/data', {
  signal: controller.signal
})
.then(response => response.json())
.catch(err => {
  if (err.name === 'AbortError') {
    console.log('Fetch 被手动中止了');
  } else {
    console.error('其他错误:', err);
  }
});

// 比如 2 秒后取消请求
setTimeout(() => controller.abort(), 2000);
```

---

### 3. `Axios` 中的用法

`Axios` 从 `v0.22.0` 版本开始，正式弃用了之前的 `CancelToken`，改用标准的 `AbortController`。

```js
const controller = new AbortController();

axios.get('https://api.example.com/data', {
  signal: controller.signal
})
.catch(err => {
  if (axios.isCancel(err)) {
    console.log('Axios 请求已取消:', err.message);
  } else {
    // 处理正常错误
  }
});

// 触发中止
controller.abort();
```

---

### 4. 原始 `AJAX` (`XMLHttpRequest`) 的用法

虽然 `XMLHttpRequest` 对象本身就有 `.abort()` 方法，但它并不直接支持 `AbortController` 这种**外部遥控**模式。

为了统一代码风格，我们通常会手动监听信号。

```js
const controller = new AbortController();
const xhr = new XMLHttpRequest();

xhr.open('GET', 'https://api.example.com/data');

// 关键步骤：监听信号的 abort 事件
controller.signal.addEventListener('abort', () => {
  xhr.abort();
  console.log('AJAX 请求已中止');
});

xhr.send();

// 触发中止
controller.abort();
```

---

### 5.对比

| 方式 | 中止手段 | 是否支持 `AbortController` |
| :--- | :--- | :--- |
| **`Fetch`** | 配置项传入 `signal` | **原生支持** |
| **`Axios`** | 配置项传入 `signal` | **原生支持**（新版本） |
| **`AJAX`** | 调用 `xhr.abort()` | **需手动封装监听** |

#### 为什么需要它？

- **切换页面**：用户离开当前页面时，取消没跑完的请求，节省性能。
- **搜索联想**：用户输入过快时，取消上一次还没返回结果的请求（防抖的进阶版）。
- **超时控制**：利用 `setTimeout` 配合 `abort()` 实现自定义超时逻辑。

### 6.AbortController的深层实现原理

要理解为什么请求会**停止**，我们得拆开 **`JavaScript` 引擎**和 **浏览器内核**（处理网络请求的底层）之间的那堵墙。

我们可以把 `AbortController` 的工作原理分为三个层面来理解：

- 信号传递机制
- 内部状态切换
- 底层资源释放

---

#### 1. 信号传递机制：遥控器与接收器

`AbortController` 本质上是一个 **发布/订阅** 模式的实现。

- **`AbortController`（遥控器）**：负责发射信号。它唯一的任务就是把内部的状态从 `false` 变成 `true`。
- **`AbortSignal`（接收器）**：负责携带信号。它是只读的，它被传给 `fetch` 或 `axios`。

当你在代码中执行 `controller.abort()` 时，发生了以下物理过程：

1. `controller` 内部的 `signal.aborted` 属性由 `false` 变为 `true`。
2. `signal` 对象立即触发一个名为 `abort` 的事件。

---

#### 2. 为什么请求会停止？

这是大家最困惑的地方：**只是改了一个变量，为什么网速极快的二进制数据流会掐断？**

##### 对于 `Fetch API`（现代原生支持）

在浏览器底层（`C++` 编写的网络模块）中，`fetch` 函数并不是一个简单的 `JS` 函数，它会开启一个 **渲染进程与浏览器主进程** 之间的通信。

1. **注册监听**：当你执行 `fetch(url, { signal })` 时，浏览器底层会在这个 `signal` 上悄悄注册一个**监听器**。
2. **瞬间联动**：一旦你调用 `abort()`，`signal` 触发事件，浏览器底层的网络小组收到通知。
3. **强行关闭句柄**：浏览器会立即关闭对应的 **`TCP` 连接** 或 **`HTTP/2` 流**。它不再等待数据下载，直接丢弃已经进入缓冲区的数据，并释放内存。

##### 对于 `XMLHttpRequest`（老式手动支持）

如上面说的，`xhr` 本身不懂 `signal`。

- **手动钩子**：你写的 `signal.addEventListener('abort', () => xhr.abort())` 就是那个手动搬动开关的人。
- **内部清理**：当执行 `xhr.abort()` 时，`XMLHttpRequest` 对象会进入 `UNSENT` 状态，停止所有网络活动。

---

#### 3. 运行时的状态机变化

我们可以通过这个简单的表格看看 `AbortSignal` 内部发生了什么：

| 时间点 | `signal.aborted` 状态 | 事件监听器 | 浏览器行为 |
| :--- | :--- | :--- | :--- |
| **请求开始** | `false` | 等待中 | 建立 `TCP` 连接，发送 `HTTP` 报文 |
| **按下 `abort()`** | **`true`** | **触发 `'abort'` 事件** | **立即关闭底层 `Socket` 连接** |
| **请求后续** | `true` | 已销毁 | `JS` 抛出 `AbortError` 异常 |

---

#### 4. 深度细节：为什么它比手动忽略结果好？

相信大家会有这样得有疑问：我不理会返回的结果不就行了吗？

其实 `AbortController` 做的远比 **忽略结果** 要多：

1. **节省带宽**：如果是下载一个 `100MB` 的文件，`abort()` 会直接断开连接，剩下的 `90MB` 就不会再下载了。
2. **释放内存**：不再需要开辟缓冲区（`Buffer`）来存放即将到来的数据。
3. **解开阻塞**：浏览器对同一个域名的并发连接数有限制（通常是 `6` 个）。及时 `abort` 掉没用的请求，可以给新的请求腾出位置。

---

#### 5. 稍微复杂一点的代码示例

如果你想看到 `signal` 内部状态的变化，可以运行这段代码：

```js
const controller = new AbortController();
const signal = controller.signal;

console.log('初始状态:', signal.aborted); // false

// 监听信号变化
signal.addEventListener('abort', () => {
  console.log('信号已变红！当前的 reason 是:', signal.reason);
});

// 2秒后按下开关
setTimeout(() => {
  controller.abort('用户不想等了'); // 可以传递自定义原因
  console.log('触发后状态:', signal.aborted); // true
}, 2000);
```

### 总结

`AbortController` 是 `JS` 暴露给你的一个 **紧急停止按钮**。

它的魔力不在于 `JS` 语法，而在于 **浏览器引擎愿意听从这个信号**，并直接从底层物理链路（`TCP`/`UDP`）上切断数据的流动。
