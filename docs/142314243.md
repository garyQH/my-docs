## 为什么reactive的解构会失效？

这是一个非常经典 的问题。

解构导致响应式丢失的原因：可以归为 **"值传递" vs "引用传递"**，以及 **"`Proxy` 拦截不到基本类型"** 这两个点上。

```js
const state = reactive({ count: 0 })
let { count } = state;
count = 1;
```
让我们拆解一下 `let { count } = state` 发生时，底层到底做了什么：

### 1. 解构本质上是赋值

当你写 `let { count } = state` 时，`JavaScript` 引擎实际上做的是：`let count = state.count;`

此时会发生两件事：

1. **触发 `Getter`**：`Vue` 的响应式系统看到你访问了 `state.count`，它会进行一次依赖收集（知道你在用这个值）。
2. **值拷贝**：
   - 如果 `state.count` 是一个**基本类型**（如数字 `0`），变量 `count` 就直接拿到了数字 `0` 的**副本**。
   - 此时，变量 `count` 只是内存里的一个普通数字，它和原来的 `state` 对象之间没有任何联系了。

### 2. `Vue` 的响应式去哪了？

- **`state`** 是一个 `Proxy` 对象，上面具有（`Getter`/`Setter`）。
- **`count`** 是解构出来的一个普通变量。当你修改 `count = 1` 时，你只是修改了一个局部变量，并没有触发 `state` 对象上的 `Setter`。
- 既然没有触发 `state` 的 `Setter`，`Vue` 就不知道数据变了，也就不会去更新页面。

---

### 3. 形象类比：吸管与果汁

想象 `state` 是一瓶装满果汁的**瓶子（对象）**，`count` 是里面的**果汁（数据）**。

- **响应式**：`Vue` 往瓶口插了一根**智能吸管**。只要你通过这根吸管喝果汁（访问属性）或往里加果汁（修改属性），智能吸管都能感知到并记录下来。
- **解构**：你把瓶里的果汁倒了一杯出来放在桌子上（`let { count } = state`）。
- **断裂**：现在你对着这杯果汁（局部变量 `count`）加糖或倒掉，**瓶口那根智能吸管（`Proxy`）是完全感知不到的**。

---

### 4. 解决方法：如何解构还能保持响应式？

如果你确实需要解构，但又不想丢失响应性，`Vue 3` 提供了专门的工具函数：`toRefs` 或 `toRef`。

#### 方法 A：使用 `toRefs` (推荐)

`toRefs` 会把 `reactive` 对象里的每一个属性都包装成一个 `ref`。

```js
import { reactive, toRefs } from 'vue'

const state = reactive({ count: 0 })

// toRefs 会把 state.count 变成一个 ref，它的 .value 指向 state.count
const { count } = toRefs(state)

// 现在修改 count.value，会触发 state.count 的修改，页面会更新
count.value++ 
```

#### 为什么 `toRefs` 行得通？

因为 `toRefs` 创建了一组"代理对象"（`refs`）。当你访问 `count.value` 时，这个 `ref` 内部实际上是去操作 `state.count`。它像是在杯子和瓶子之间连接了一根**透明软管**，保持了同步。

---

> **这也就是为什么 `Vue` 官方文档里反复强调：不要对 `reactive` 对象进行普通解构。**

---

### 5.如果解构出来的是一个引用类型呢？

答：如果你解构出来的是一个**对象（引用类型）**，情况会变得非常有趣：**响应式并不会完全消失，但它会变得"半身不遂"。**

举个栗子拆解这个现象：

#### 1. 结论：由于"引用传递"，响应式依然存在

在 `JavaScript` 中，对象是按**引用**传递的。当你解构一个嵌套对象时，你拿到的变量依然指向原始内存中的那个对象。

```js
const state = reactive({
  user: { name: 'Alice', age: 25 },
  count: 0
})

// 解构出一个对象
let { user } = state 

// 此时：user 变量指向的是 state 内部的那个对象内存地址
```

#### 2. 为什么说它"半身不遂"？

##### 情况 A：修改对象的属性（有效）

- 如果你执行 `user.name = 'Bob'`，页面**会更新**。

**原因**：`user` 指向的地址和 `state.user` 是一样的。由于 `state` 是 `reactive` 的，`Vue` 已经深度代理了其内部的所有层级。当你修改 `user.name`，实际上触发了那个深度代理对象的 `Setter`。

##### 情况 B：直接替换解构出来的变量（失效）

如果你执行 `user = { name: 'Charlie' }`，页面**不会更新**。

- **原因**：你只是让 `user` 这个局部变量指向了一个新对象，而 `state.user` 依然指向旧对象。

---

#### 3. 核心区别：基本类型 vs 对象类型

为了让你一眼看穿本质，我们对比一下这两种解构：

| 解构对象 | 代码示例 | 修改属性 (`obj.a = 1`) | 直接替换变量 (`obj = ...`) |
| :--- | :--- | :--- | :--- |
| **基本类型** | `let { count } = state` | 根本没有属性可以改 |  失去响应式 |
| **对象类型** | `let { user } = state` | **依然有效** |  失去响应式 |

---

#### 4. 深度原理：`Vue` 的递归代理

为什么修改 `user.name` 能成功？

因为当你创建 `reactive(state)` 时，`Vue` 会进行**深度响应式转换**。它不仅把 `state` 变成了 `Proxy`，当你访问 `state.user` 时，它也会确保返回的是一个 `Proxy`。

所以，解构出来的 `user` 变量，它本身就是一个被 `Vue` 包装过的 **`Proxy` 对象**。只要你通过这个 `Proxy` 去改它内部的属性，`Vue` 就能抓到你。

#### 5. 最佳实践建议

虽然解构对象能保留一部分响应式，但在 `Vue 3` 开发中，我们仍然**强烈建议不要这样做**，原因有二：

1. **心智负担**：你得时刻记住哪些能改（属性），哪些不能改（变量本身）。
2. **代码一致性**：为了防止手抖直接赋值，始终使用 `toRefs` 才是最稳妥的做法：

```js
// 这样做，无论 user 还是 count，都变成了 ref，逻辑统一且安全
const { user, count } = toRefs(state)

user.value = { name: 'Bob' } // 连整体替换都能触发更新了！
```
