# 微前端应用

随着前端应用规模的爆发式增长，**微前端**（`Micro Frontends`）已成为解决**巨石应用**维护困境的主流方案。

它核心解决的是**技术栈无关**、**独立开发部署**以及**增量升级**的问题。

---

## 一、 深度解析：什么是微前端？

微前端并非一种特定的技术，而是一种**架构思想**。

它将后端微服务的理念引入到前端，把一个大型 `Web` 应用拆分为若干个可以**独立运行、独立开发、独立部署**的小型应用，最后通过一个 **主应用（基座）** 进行聚合。

### 核心价值

1. **技术栈无关**：主应用不关心子应用是用 `Vue`、`React` 还是 `Angular` 开发的。
2. **独立性**：每个团队负责自己的子应用，发布时无需全量回归测试。
3. **遗留系统迁移**：可以把老旧的 `jQuery` 或 `AngularJS` 项目作为一个子应用嵌入，实现平滑过渡。

### 关键技术难点

- **路由分发**：主应用如何根据 `URL` 准确切换子应用。
- **应用隔离**：防止子应用之间的 `JS` 全局变量污染和 `CSS` 样式冲突。
- **通信机制**：父子应用、兄弟应用之间如何高效传递数据。

---

## 二、`qiankun` vs `Webpack` 模块联邦（`Module Federation`）

这是目前市面上最火的两种实现思路，但它们的底层逻辑完全不同。

### 1. `qiankun`（基于 `Single-SPA` 的包装）

**原理**：通过 `HTML Entry` 模式加载子应用。

主应用像请求网页一样拉取子应用的 `HTML`，解析出 `JS` 和 `CSS` 并执行。

| 优点 | 缺点 |
| :--- | :--- |
| **开箱即用**：封装了沙箱（`Sandbox`）、样式隔离、预加载等功能。 | **性能损耗**：由于需要解析 `HTML` 和代理沙箱，存在一定的运行时开销。 |
| **技术侵入性低**：对现有代码改造极小。 | **隔离限制**：虽然有沙箱，但在某些复杂场景下（如多实例）仍有坑。 |
| **生态成熟**：文档丰富，社区填坑多。 | **依赖重复**：默认情况下，每个子应用都会携带自己的基础库（如 `Vue`/`React`）。 |

### 2. `Module Federation`（`Webpack 5` 新特性）

**原理**：允许一个 `JavaScript` 应用在运行时动态地加载另一个应用的代码。它不再是以**页面**为单位，而是以**模块**为单位。

| 优点 | 缺点 |
| :--- | :--- |
| **共享依赖**：多个应用可以共享同一个 `React` 或 `Vue` 实例，极大减小体积。 | **环境限制**：强依赖 `Webpack 5`，`Vite` 项目使用较困难。 |
| **去中心化**：没有严格的**基座**概念，每个应用既可以是提供方也可以是使用方。 | **缺乏隔离**：默认没有提供 `CSS` 隔离和 `JS` 沙箱，需要开发者手动处理污染。 |
| **极致性能**：运行时几乎没有额外开销，像加载本地模块一样快。 | **配置复杂**：对版本管理和依赖共享的配置要求较高。 |

---

## 三、 大厂主流实现方案

不同大厂根据业务场景，往往会自研或在开源方案上做重度定制：

### 1. 蚂蚁金服 / 阿里：中心化基座模式（`qiankun`）

这是最典型的方案。通过一个 `Main Framework` 作为基座，统一管理路由、权限、用户信息和子应用生命周期。

- **适用场景**：大型企业级中后台系统（如云控制台、`OA` 系统）。

### 2. 字节跳动 / 腾讯：无感微前端（`Garfish` / `Hel-Micro`）

字节跳动的 `Garfish` 提供了更强大的沙箱隔离和资源预加载能力。

腾讯的一些团队则偏向使用类似 `Hel-Micro` 的远程模块注入方案，更强调**模块级**的复用，类似于把微前端玩成了动态 `NPM` 包。

### 3. 美团 / 百度：基于 `Web Components` 的隔离

部分团队利用原生浏览器能力 `Web Components`（如 `Shadow DOM`）来实现完美的样式隔离。

这种方案的兼容性要求较高，但在现代浏览器环境下最为轻量且原生。

### 4. 混合方案（`The Hybrid Way`）

许多大厂在实际操作中会结合两者：

- **外层**：使用 `qiankun` 管理大的业务模块，确保独立部署和样式隔离。
- **内层**：使用 `Module Federation` 共享基础组件库（`UI Library`）或公共工具类，减少包体积。

---

## 四、 总结：如何选择？

- 如果是**存量老项目改造**，或者团队技术栈杂乱，选 `qiankun`。它够稳，帮你处理了 90% 的隔离难题。
- 如果是**纯新项目**，追求极致性能，且全员 `Webpack 5`/`Vite` 阵营，可以尝试 `Module Federation`，它代表了未来模块共享的趋势。

> **提示**：微前端会带来部署复杂度（`CI/CD`）的提升和调试的困难。如果项目还没大到需要 10 个以上前端同学协同，通常建议先保持单体架构。

看一看具体的 `Webpack 5` 模块联邦配置代码，或者了解如何在 `Vue 3` 项目中快速接入 `qiankun` 吗？

## 五、`Garfish`、`Hel-Micro` 与 `Web Components` 方案深度对比

我们来深度对比一下字节跳动的 `Garfish`、腾讯的 `Hel-Micro`，以及美团/百度推崇的 `Web Components`（以无界 `wujie` 为代表）方案。

这些方案代表了微前端演进的三个不同方向：**极致工程化、去中心化模块化、以及利用浏览器原生的硬隔离。**

### 1. 字节跳动：`Garfish`（极致工程化与增强沙箱）

`Garfish` 在原理上与 `qiankun` 类似，都是基于 `HTML Entry`，但它在**大型复杂业务**的细节处理上做得更深。

- **核心原理**：
  - `Loader` **拦截**：它自研了一套资源加载器，不仅能解析 `HTML`，还能对资源进行分级预加载（`Preload`/`Prefetch`），甚至支持子应用资源的静态替换。
  - **副作用劫持**（`Side Effect`）：除了 `window` 代理，它对定时器、事件监听、`localStorage` 等浏览器原生 `API` 做了极其细致的劫持，确保子应用卸载时能彻底**净身出户**。
- **细节差异**：
  - **多实例支持**：比 `qiankun` 更早且更稳地支持同一个页面挂载多个子应用实例。
  - **调试体验**：自带调试面板，能清晰看到子应用加载状态、沙箱快照对比等。
- **对比总结**：如果说 `qiankun` 是微前端的**通用模板**，`Garfish` 就是针对**超大规模、对性能和隔离有洁癖**的业务设计的进阶工具。

### 2. 腾讯：`Hel-Micro`（去中心化的运行时模块共享）

`Hel-Micro` 的思路完全不同，它不追求**应用**级别的加载，而是追求**模块**级别的远程复用。

- **核心原理**：
  - **远程模块 SDK 化**：它将模块联邦（`Module Federation`）的思想进行了 `SDK` 化。子应用不需要 `Webpack 5`，只要按照 `Hel` 协议打包，就能变成一个远程可调用的**云端模块**。
  - **运行时导入**：在代码里通过如下方式直接调用另一个应用的组件或函数：

```js
await importApp('app-name')
```

- **细节差异**：
  - **免构建升级**：子应用更新代码后，主应用不需要重新打包，刷新浏览器直接拉取最新模块。
  - **类型支持**：它能在运行时提供远程模块的 `TypeScript` 类型定义，解决了微前端开发中**盲调**的痛点。
- **对比总结**：它更像是一个**跨项目的运行时** `NPM`。适合那种**你有我，我有你**、频繁需要互相调用组件和逻辑的协作场景。

### 3. 美团/百度/京东：`Web Components` & `Iframe`（原生隔离派）

这类方案以京东的**无界**（`wujie`）为典型代表，它解决了 `qiankun` 最难处理的 `CSS` 隔离和弹窗问题。

- **核心原理**：
  - `JS` 在 `Iframe` 跑，`CSS`/`DOM` 在 `Web Component` 跑：
    1. 它创建一个隐藏的 `iframe` 来运行子应用的 `JS`，利用 `iframe` 天然的物理隔离，`JS` 沙箱稳如泰山。
    2. 它在主应用中创建一个 `Web Component`（`Shadow DOM`），将子应用的 `DOM` 结构丢进去。
  - **代理连接**：通过代理机制，让 `iframe` 里的 `JS` 操作的 `DOM` 实际上指向主页面的 `Web Component`。
- **细节差异**：
  - **样式物理隔离**：`Shadow DOM` 是浏览器原生的样式屏障，外面的样式进不去，里面的样式出不来。
  - **保活**（`Keep-alive`）：由于 `JS` 运行在特殊的 `iframe` 中，切换应用时只需隐藏 `DOM`，`JS` 状态可以完美保留。
- **对比总结**：这是目前**最省心、兼容性最强**（只要不考虑 `IE`）的方案。子应用几乎 `0` 改造，且完全不用担心全局变量和样式冲突。

## 六、综合对比表

| 特性 | `qiankun`（阿里） | `Garfish`（字节） | `Hel-Micro`（腾讯） | 无界/`Web Components` |
| :--- | :--- | :--- | :--- | :--- |
| **隔离技术** | `Proxy` 代理沙箱 | 增强版代理沙箱 | 基本无隔离 (需自备) | `Iframe`（`JS`）+ `ShadowDOM`（`CSS`） |
| **接入成本** | 中 (需改生命周期) | 中 | 高 (需符合 `SDK` 规范) | **极低**（甚至无需改造） |
| **性能开销** | 中 (解析 `HTML`) | 中 | **极低**（原生加载） | 低（主要在首次初始化） |
| **样式安全** | 较弱 (容易被穿透) | 中 | 无 | **极强**（原生屏障） |
| **适用场景** | 中后台系统 | 大型集团级复杂应用 | 跨团队高频功能复用 | **追求稳定隔离、保活、老项目** |

## 七、结论：我该选哪个？

1. **如果你追求稳定、成熟、文档多**：选 `qiankun`。
2. **如果你是超大型组织，有复杂的性能优化需求**：研究 `Garfish`。
3. **如果你的子应用之间需要像调用库一样互相调代码**：选 `Hel-Micro`。
4. **如果你被样式冲突、弹窗错位折磨疯了，想找最简单的方案**：强烈推荐 **无界**（`wujie`）。
