## Vue3的 Provide 和 Inject

在 `Vue 3` 中，**`Provide`** 和 **`Inject`** 是一种被称为 **"依赖注入"** 的模式。

如果把父子组件传参（`Props`）比作"接力棒"，那么 `Provide` / `Inject` 就像是组件树里的 **"广播站"**。

---

### 1. 为什么要用它？（解决"深层嵌套"痛苦）

想象你有一个极其庞大的组件树，根组件的数据要传给第五层的一个小组件：

- **用 `Props`**：你必须像传声筒一样，在中间的第 1、2、3、4 层组件里全部写一遍 `Props`。即便这些中间组件根本不需要这些数据，它们也得当"搬运工"。这就是所谓的 **`Props Drilling`（属性钻取）**。
- **用 `Provide`/`Inject`**：根组件直接"广播"数据，第五层组件直接"收听"，中间那四层完全可以当做没这回事。

---

### 2. 基本用法

#### 第一步：在祖先组件 `Provide`

```js
import { provide, ref } from 'vue'

export default {
  setup() {
    const themeColor = ref('royalblue')
    
    // provide(钥匙, 内容)
    provide('theme', themeColor) 
  }
}
```

#### 第二步：在后代组件 `Inject`

```js
import { inject } from 'vue'

export default {
  setup() {
    // inject(钥匙, 默认值)
    const theme = inject('theme', 'gray')
    
    return { theme }
  }
}
```

---

### 3. 如何保持响应式？

这是最关键的一点：**如果你提供的是一个 `ref` 或 `reactive` 对象，那么注入方拿到的也是响应式的。**

这意味着：祖先组件里的内容变了，孙子组件里的页面会 **自动更新**。

---

### 4. 修改数据的单向数据流

虽然孙子组件可以直接改数据，但这会让调试变得非常困难（你不知道是谁改了数据）。

**最佳实践**：遵循 **谁提供，谁负责修改** 的原则。

祖先组件提供数据时，顺便提供一个 **修改函数**。

```js
// 祖先组件
const location = ref('North Pole')
const updateLocation = () => {
  location.value = 'South Pole'
}

provide('location', {
  location, // 数据
  updateLocation // 修改的方法
})
```

---

### 5. `Provide` / `Inject` vs. `Vuex` / `Pinia`

你可能会问："这不就是全局状态管理吗？" 

它们还是有区别的：

- **`Vuex` / `Pinia`**：是 **全站级别** 的，任何组件都能访问，通常用于用户信息、登录状态等。
- **`Provide` / `Inject`**：是 **组件树级别** 的。只有作为这个 **广播源** 的后代才能收到。常用于编写复杂的**组件库**（比如 `Select` 组件提供数据给内部的 `Option` 列表）。

---

### 总结

- **`Provide`**：祖先发货。
- **`Inject`**：后代收货。
- **核心优势**：跨层级，摆脱 `Props` 噩梦。
- **核心注意**：为了代码健壮性，建议配合 `readonly` 使用（防止后代乱改），或显式提供修改函数。
