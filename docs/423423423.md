## ref的设计原因

> **简单来说：** **`Proxy` 的限制决定了 `ref` 的实现形式。**

---

### 1. `Proxy` 只能拦截对象

`Vue 3` 的响应式核心是 `Proxy`。

而`Proxy` 只能代理**引用类型**（对象、数组、`Map`、`Set` 等）。

对于基本类型（`String`、`Number`、`Boolean`、`null`、`undefined`），`JavaScript` 引擎不允许对其建立代理。

### 2. 包装成对象是为了追踪

为了让基本类型也具备响应式，`Vue` 必须给它找一个 **外壳**。

- 如果你直接传递一个数字 `let count = 0`，`Vue` 根本无法监控到这个变量的变化。
- 通过将其包装成一个对象 `{ value: 0 }`，`Vue` 就可以对这个对象的 **`.value` 属性** 进行拦截。

### 3. 引用传递的问题

在 `JavaScript` 中，基本类型是**值传递**，而对象是**引用传递**。

- 如果 `ref` 不包装成对象，当你把一个响应式的字符串传给组件或函数时，它会丢失响应性（变成了一个普通的字符串副本）。
- 包装成对象后，由于传递的是引用，无论这个对象被传递到哪里，大家操作的都是**同一个 `.value` 属性**，`Vue` 就能始终追踪到它的变化并触发视图更新。

### 4. 为什么 key 必须叫 value？

这其实是为了**一致性**。

虽然技术上 `Vue` 可以叫它 `{ data: ... }` 或 `{ content: ... }`，但统一使用 `.value` 可以让开发者在使用不同的 `ref` 时形成固定的心智模型。

此外，`Vue` 在模板（`template`）中会自动帮你处理 `.value`（自动脱靶/`unwrapping`），这要求内部结构必须是标准化的。

---

### 总结

- **`Proxy` 限制**：无法直接代理原始值。
- **包装方案**：将原始值挂载到对象的 `.value` 属性上。
- **响应式原理**：当你访问或修改 `xxx.value` 时，`Vue` 触发 `get` 和 `set` 拦截，从而收集依赖或派发更新。
- 实际在底层 `ref` 也并非使用了 `Proxy`。

**对比 `reactive`：**

| 特性 | `ref` | `reactive` |
| :--- | :--- | :--- |
| **参数类型** | 原始值 + 对象 | 仅限对象 |
| **访问方式** | 需加 `.value`（`JS` 中） | 直接访问 |
| **底层实现** | 包装成带 `.value` 的 `Proxy` 对象 | 直接对原对象进行 `Proxy` 包装 |
