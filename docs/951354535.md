# Vite

`Vite`（法语意为 **快速**）是当前前端生态中统治级的构建工具。

它由 **尤大** 开发，彻底改变了 `Web` 开发的启动和热更新体验。

要深入理解 `Vite` 为什么快，不能只持有 **它不用打包** 这一个观点。

`Vite` 的快源于它对**开发环境**和**生产环境**采用了完全不同的策略。

我们需要从 **开发环境的冷启动**、**模块解析逻辑**、**热更新机制**（`HMR`）以及 **底层语言优势** 四个维度来拆解。

---

## 1. 彻底改变启动策略：`No-Bundle` vs `Bundle`

这是 `Vite` 与 `Webpack` 最本质的区别。

### 传统工具（如 `Webpack`/`Rollup`）：先打包，后启动

传统的构建工具是 **束手束脚** 的。

在浏览器看到页面之前，`Webpack` 必须从 `main.js` 入口开始，递归地解析所有 `import`，构建一个巨大的依赖图（`Dependency Graph`），然后将它们转换、合并、压缩，最后输出一个 `bundle.js`。

- **痛点**：随着项目规模变大（比如有数千个模块），这个**构建图**的过程会呈线性增长，导致启动需要几十秒甚至几分钟。

### `Vite`：先启动，后按需加载

`Vite` 完全抛弃了 **打包** 的概念。

它直接启动一个本地服务器，并利用现代浏览器原生支持的 **ES Modules**（`ESM`）属性（即 `<script type="module">`）。

- **按需编译**：当浏览器请求某个 `URL` 时（例如 `import './App.vue'`），`Vite` 拦截这个请求，并在服务器端即时对该文件进行编译（如将 `Vue` 模板转为 `JS`），然后返回给浏览器。
- **结果**：无论你的项目有 10 个模块还是 10,000 个模块，`Vite` 的**启动时间几乎是常数级**的（通常在 300ms 以内），因为它根本不预处理没被请求的代码。

---

## 2. 依赖预构建：`esbuild` 的降维打击

虽然 `Vite` 提倡 `ESM`，但有两个现实问题：

1. `CommonJS` **兼容性**：很多老牌 `NPM` 包（如 `React`）依然是 `CJS` 格式，浏览器无法识别。
2. **网络请求压力**：有些库（如 `lodash-es`）内部有几百个小文件，如果浏览器直接请求，几百个 `HTTP` 请求会造成严重的网络拥塞。

`Vite` 的**解决方案：依赖预构建**（`Dependency Pre-bundling`）

`Vite` 在初次启动或依赖变更时，会扫描你的代码，找出所有 `node_modules` 里的依赖，使用 `esbuild` 将它们统一预打包成高性能的 `ESM` 模块。

- **为什么用** `esbuild` **？**
  - `esbuild` 是用 **Go 语言**编写的。传统的 `JS` 构建工具运行在 `Node.js` 上，受限于 `JS` 的解释执行速度。而 `Go` 是编译型语言，且 `esbuild` 充分利用了多核 `CPU` 并行处理。
- **速度对比**：在处理相同的重型依赖转换任务时，`esbuild` 的速度通常是 `Webpack` 的 **10-100 倍**。

---

## 3. 极速的热更新（`HMR`）

在 `Webpack` 中，如果你修改了一个文件，`Webpack` 会尝试重新计算依赖图并**热替换**模块，但随着项目增大，由于内部闭包和依赖链的复杂性，`HMR` 的延迟会越来越明显。

### `Vite` 的 `HMR` 实现

1. **基于** `ESM` **的解耦**：`Vite` 的 `HMR` 是在原生 `ESM` 之上实现的。当文件改变时，`Vite` 只需要使该模块在浏览器端的缓存失效。
2. **利用浏览器缓存**：
   - **源码模块**：`Vite` 利用 `HTTP` 协商缓存（`304 Not Modified`）来减少不必要的代码传输。
   - **依赖模块**：预构建的依赖（如 `Vue`、`Axios`）会被设置为强缓存（`Cache-Control: max-age=31536000, immutable`），一旦构建完成，除非版本变动，否则浏览器永远不会重复请求。

---

## 4. 总结：快在哪？

| 环节 | `Webpack` / 传统工具 | `Vite` |
| :--- | :--- | :--- |
| **启动原理** | 递归解析依赖、打包、启动服务器 | 启动服务器、浏览器按需请求、动态编译 |
| **底层语言** | `JavaScript`（`Node.js`） | `Go`（`esbuild`）进行重型预处理 |
| `HMR` **效率** | 随项目规模变大而变慢 | 始终保持极快，与规模无关 |
| **缓存利用** | 主要依靠磁盘缓存（`Filesystem Cache`） | 深度利用 `HTTP` 缓存（强缓存 + 协商缓存） |

---

### 思考：

> 既然 `Vite` 这么快，为什么生产环境还要用 `Rollup` 打包，不直接用 `ESM` 部署？

这是因为在生产环境下：

1. **嵌套导入问题**：未经打包的 `ESM` 在网络环境下会导致大量请求，严重的**瀑布流**加载会影响首屏速度。
2. **优化空间**：`Rollup` 在 `Tree-shaking`（**摇树优化**）、`Code Splitting`（**代码分割**）和 `CSS` **注入优化**方面比目前的原生 `ESM` 部署方案更成熟。

## 5.常用配置（`vite.config.ts`）

以下是一个涵盖 80% 业务场景的配置模板：

```js
import { defineConfig, loadEnv } from 'vite'
import vue from '@vitejs/plugin-vue'
import path from 'path'

export default defineConfig(({ mode }) => {
  // 根据当前环境（development/production）加载环境变量
  const env = loadEnv(mode, process.cwd())

  return {
    plugins: [vue()], // 插件配置
    resolve: {
      alias: {
        '@': path.resolve(__dirname, './src'), // 设置路径别名
      },
    },
    server: {
      port: 3000,
      open: true,
      proxy: {
        '/api': {
          target: env.VITE_API_URL, // 代理接口地址
          changeOrigin: true,
          rewrite: (path) => path.replace(/^\/api/, ''),
        },
      },
    },
    build: {
      outDir: 'dist',
      sourcemap: false,
      chunkSizeWarningLimit: 1500, // 调整包大小警告限制
      rollupOptions: {
        output: {
          // 静态资源分库打包，优化缓存
          manualChunks(id) {
            if (id.includes('node_modules')) {
              return 'vendor';
            }
          },
        },
      },
    },
  }
})
```

## 6.总结问题

### 1. `Vite` 和 `Webpack` 的区别？

- **构建思路**：`Webpack` 是全量打包，`Vite` 开发环境是不打包（`No-bundle`），只做按需编译。
- **启动速度**：`Vite` 启动速度不随项目体积线性增长；`Webpack` 随体积增长明显变慢。
- **热更新**：`Vite` 基于浏览器缓存和 `ESM` 链路，速度极快且稳定；`Webpack` 需重新构建部分依赖链路。
- **成熟度**：`Webpack` 插件生态极其丰富，适配各种老旧项目；`Vite` 更现代，但在处理极其复杂的特殊构建需求时不如 `Webpack` 灵活。

### 2. 什么是依赖预构建？为什么要用 `esbuild`？

- **原因**：（1）转换非 `ESM` 模块（如 `CJS`）；（2）合并请求（如将几百个 `lodash` 小文件合并为一个）。
- `esbuild`：因为 `esbuild` 是用 `Go` 写的，编译速度比 `JS` 快几个数量级，适合这种只追求转换速度的任务。

### 3. `Vite` 的热更新原理？

当文件变动时，`Vite` 服务器通过 `WebSocket` 通知浏览器。浏览器根据 `import.meta.hot` 的接口定义，仅重新请求发生变化的模块。`Vite` 会在返回的响应头中利用 `HTTP` 缓存（协商缓存 `304` 或强缓存）来进一步加速。

### 4. 为什么生产环境不用 `esbuild` 而是 `Rollup`？

虽然 `esbuild` 很快，但目前它在**代码分割**（`Code Splitting`）和 `CSS` 处理、`Tree-shaking` 等生产环境核心功能上还不够成熟稳定。`Rollup` 拥有更成熟的插件生态和更优秀的构建产物。

### 5. 如何处理 `Vite` 环境变量？

- 使用 `.env` 文件。
- 变量必须以 `VITE_` 开头才能在代码中通过 `import.meta.env.VITE_XXX` 访问。
- 原理：`Vite` 使用 `dotenv` 解析文件，并通过定义全局常量的方式注入到代码中。
