# Sass、Scss、Less、PostCSS

在前端开发中，这四个工具经常被提到，但它们的角色完全不同。

我们可以把它们分为两类：**预处理器** 和 **后处理器**。

---

## 一、核心区别与关系

### 1. 预处理器：`Sass`/`Scss` 和 `Less`

它们的作用是**扩展** `CSS` **的语法**。

由于原生 `CSS` 长期缺乏变量、嵌套和函数，这些工具允许你用类似编程的方式写样式，然后**编译**成普通的 `CSS`。

- `Sass` **与** `Scss` **的对比：**
  - `Sass`（`Syntactically Awesome Style Sheets`）：最早的版本，采用缩进语法，**不需要大括号和分号**（类似 `Python`）。
  - `Scss`（`Sassy CSS`）：`Sass` 3.0 引入的新语法，**完全兼容** `CSS` **语法**（有大括号和分号），是目前业界的主流。
- `Less`：受 `Sass` 启发，语法与 `Scss` 非常接近。
  - **区别**：`Less` 是基于 `JavaScript` 的，可以在浏览器端运行（虽然不推荐）；`Sass`/`Scss` 早期基于 `Ruby`，现在主要基于 `Dart`。`Sass` 的逻辑处理能力（循环、条件判断）通常比 `Less` 更强大。

### 2. 后处理器：`PostCSS`

`PostCSS` 本身**不改变语法**，它是一个**插件平台**。

它将 `CSS` 解析成抽象语法树（`AST`），然后交给插件去处理。

- **最著名的插件**：`Autoprefixer`（自动加浏览器前缀 `-webkit-`）。
- **关系**：你可以在用完 `Sass` 编译后，再用 `PostCSS` 进行优化。`Vite` 默认内置了对 `PostCSS` 的支持。

---

## 二、代码 `Demo` 对比

假设我们要实现一个：**定义主题色、嵌套导航栏、以及自动添加浏览器前缀**的场景。

### 1. `Scss` 示例（主流推荐）

文件名：`style.scss`

```css
$primary-color: #3498db; // 定义变量

@mixin flex-center { // 定义混合器
  display: flex;
  justify-content: center;
}

.navbar {
  background-color: $primary-color;
  &:hover { background-color: darken($primary-color, 10%); } // 颜色函数

  ul { // 嵌套语法
    @include flex-center;
    list-style: none;
  }
}
```

### 2. `Sass` 示例（老式缩进）

文件名：`style.sass`

```css
$primary-color: #3498db

=flex-center
  display: flex
  justify-content: center

.navbar
  background-color: $primary-color
  &:hover
    background-color: darken($primary-color, 10%)
  
  ul
    +flex-center
    list-style: none
```

### 3. `Less` 示例

文件名：`style.less`

```css
@primary-color: #3498db; // 变量用 @

.flex-center() { // 混合器类似函数
  display: flex;
  justify-content: center;
}

.navbar {
  background-color: @primary-color;
  &:hover { background-color: darken(@primary-color, 10%); }

  ul {
    .flex-center(); // 调用
    list-style: none;
  }
}
```

### 4. `PostCSS` 示例（处理产物）

`PostCSS` 不像上面那样写，它通常处理上述工具生成的标准 `CSS`。

**输入**（标准 `CSS`）：

```css
.box {
  display: flex;
  user-select: none;
}
```

**通过** `PostCSS`（`Autoprefixer` 插件）**处理后的输出：**

```css
.box {
  display: -webkit-box;
  display: -ms-flexbox;
  display: flex;
  -webkit-user-select: none;
  -moz-user-select: none;
  -ms-user-select: none;
  user-select: none;
}
```

## 三、总结对比表

| 工具 | 类型 | 变量符号 | 逻辑能力 | 特点 |
| :--- | :--- | :--- | :--- | :--- |
| `Scss` | 预处理器 | `$` | 极强 | 生态最广，`Vite`/`Webpack` 配合首选。 |
| `Sass` | 预处理器 | `$` | 极强 | 无分号大括号，写起来快但易错。 |
| `Less` | 预处理器 | `@` | 一般 | 语法简单，上手极快，国内 `Ant Design` 等在使用。 |
| `PostCSS` | 插件平台 | 无 | 取决于插件 | 负责兼容性、压缩、未来的 `CSS` 语法降级。 |

## 四、常见问题

1. 为什么现在大家都用 `Scss` 而不是 `Sass`？
   - `Scss` 兼容 `CSS` 原生写法，学习成本低，现有 `CSS` 项目可以直接改名 `.scss` 运行。
2. `Scss` 和 `Less` 的变量提升区别？
   - `Less` 的变量有 **作用域提升** 概念，可以在声明前使用；`Scss` 则必须先声明后使用。
3. 什么是 `CSS Modules`？它和这些工具有冲突吗？
   - 不冲突。`CSS Modules` 解决的是**样式隔离**（防止类名冲突）。`Vite` 中只需将文件命名为 `index.module.scss`，`Vite` 就会自动处理。

## 五、 CSS Modules

提到 `CSS Modules`，就不得不提到另一个大家熟悉的隔离手段 `scoped`！

对于两者而言，**它们的目标一致，但实现手段和原理完全不同。**

你可以理解为：`scoped` 是 `Vue` 的**官方私有化方案**，而 `CSS Modules` 是**业界通用的工程化方案**。

### 原理大比拼

#### 1. `scoped`

**原理：属性选择器注入**

当你写 `<style scoped>` 时，`Vue Loader` 会做两件事：

1. **给** `HTML` 元素加属性：为当前组件的所有 `DOM` 节点添加一个全局唯一的自定义属性（如 `data-v-555845`）。
2. **改写** `CSS` 选择器：在你的 `CSS` 选择器末尾加上这个属性选择器。

**编译前：**

```html
<template>
  <div class="title">Hello</div>
</template>
<style scoped>
  .title { color: red; }
</style>
```

**编译后：**

```html
<div class="title" data-v-555845>Hello</div>
```

```css
.title[data-v-555845] { color: red; }
```

#### 2. `CSS Modules`（通用方案）

**原理：类名混淆**（`Class Hashing`）

`CSS Modules` 会将你的类名重新编译成一个**哈希字符串**，并生成一个 `JS` 对象来映射原始类名和哈希类名。

**编译前**（`Vite` 中使用 `style.module.css`）：

```css
/* style.module.css */
.title { color: blue; }
```

**编译后：**

```html
<div class="_title_1abc2_5">Hello</div>
```

```css
._title_1abc2_5 { color: blue; }
```

在 `JS` 中，你会通过 `styles.title` 来获取这个随机生成的字符串。

### 核心区别对比

| 特性 | `Scoped CSS` | `CSS Modules` |
| :--- | :--- | :--- |
| **控制权** | 依然使用原始类名，通过属性限制范围。 | 类名被完全改变，通过 `JS` 变量引用。 |
| **穿透问题** | 需要使用 `:deep()` 才能修改子组件样式。 | 很难直接穿透，通常需要组合类名或全局定义。 |
| **性能** | 属性选择器性能略低于类选择器。 | 纯类名选择器，性能最优。 |
| **显性度** | 隐式处理，写起来最省事（`Vue` 默认）。 | 显式处理，需要在 `HTML` 中绑定变量。 |
| **跨框架** | 仅限 `Vue`（或类似语法的框架）。 | `React`、`Vue`、`Angular` 甚至原生 `JS` 都能用。 |

## 六、深度选择器 `:deep()`

由于 `scoped` 是通过属性选择器工作的，如果你引用了一个第三方组件（如 `Element Plus` 的 `el-button`），你直接写 `.el-button` 是生效不了的，因为该组件内部的 `DOM` 没有你当前组件的 `data-v` 属性。

**解决方法：**

```css
/* 在 scoped 中使用穿透 */
.parent :deep(.el-button) {
  /* 编译后会变成 .parent[data-v-xxx] .el-button */
  /* 此时就能匹配到子组件内部的类名了 */
  color: green;
}
```

### `Vite` 中的用法

在 `Vite` 中，你不需要安装任何插件就能同时使用这两者：

1. `Vue` **项目**：直接在 `.vue` 文件加 `scoped` 即可。
2. 通用项目 / `SCSS`：将文件命名为 `*.module.scss`，`Vite` 会自动识别并开启 `CSS Modules`。

```js
// 在 Vite/React 中引入 CSS Modules
import styles from './header.module.scss'

function Header() {
  return <div className={styles.title}>Hello</div>
}
```

### 总结：该选哪一个？

- 如果你用 `Vue`，**90%** 的场景请坚持使用 `scoped`，因为它和 `Vue` 的模板系统集成得最完美，心智负担最低。
- 如果你用 `React`，或者你希望样式逻辑高度 `JS` 化（比如根据状态动态拼凑多个混淆类名），那么 `CSS Modules` 是更好的选择。

## 七、`:deep()`的原理

> `:deep()` 本质上是一个 **挪位置** 的操作。

### 1. 默认情况：属性选择器在末尾

在 `scoped` 模式下，`Vue` 会默认把你组件的唯一标识（比如 `data-v-abc`）拼在**每一个**选择器的最后面。

**你的代码：**

```css
.parent .child { color: red; }
```

`Vue` **编译出的结果：**

```css
.parent .child[data-v-abc] { color: red; }
```

> **翻译成人话：** 浏览器会去找 `class` 为 `parent` 节点下的、且**自身带有 `data-v-abc` 属性**的 `child` 节点。因为子组件内部的节点没有这个属性，所以找不到，样式失效。

### 2. 使用 `:deep()`：属性选择器提前

当你给选择器加上 `:deep()` 时，`Vue` 编译器的逻辑变了：它会把 `data-v-abc` 放在 `:deep()` **前面**的那个选择器上。

**你的代码：**

```css
.parent :deep(.child) { color: red; }
```

`Vue` **编译出的结果：**

```css
.parent[data-v-abc] .child { color: red; }
```

> **翻译成人话：** 浏览器会去找**带有 `data-v-abc` 属性**的 `parent` 节点，然后在这个节点下面找**所有** `class` 为 `child` 的节点（不管 `child` 身上有没有属性）。

### 3. 一个直观的对比实验

假设你在开发一个页面，引入了一个第三方组件 `ThirdParty`。

| 编写方式 | 编译后的 `CSS` | 效果 |
| :--- | :--- | :--- |
| `.box .title` | `.box .title[data-v-123]` | **失败**。因为第三方组件里的 `.title` 标签身上没有 `data-v-123`。 |
| `.box :deep(.title)` | `.box[data-v-123] .title` | **成功**。只要你的父容器 `.box` 是在这个组件里的（有 `data-v-123`），它内部所有的 `.title` 都会变色。 |

### 4. 为什么要这么设计？

> **为什么不让父组件的** `data-v` **自动传给子组件的所有标签？**

- **样式污染防止**：如果父组件的属性自动渗透进子组件，那么父组件写一个简单的 `div { color: red; }` 就会把你引用的所有第三方组件（如 `Element Plus`）内部的布局全部搞乱。
- **组件封装性**：子组件的内部结构应该是黑盒。只有当你明确使用 `:deep()` 时，才代表你**蓄意**要修改人家内部的样式。

### 5. 写法演变

在 `Vue` 的不同版本或插件中，写法有所不同，但原理**完全一样**：

1. **早期**（`Sass`/`Less`）：使用 `/deep/ .child` 或 `::v-deep .child`。
2. `Vue 3` **标准**：`:deep(.child)`。

**总结一句话：**

`:deep()` 就是把 `data-v`（**通行证**）从**具体的子节点**身上，挪到了**父级容器**身上，从而让父级容器下的所有后代都能被匹配到。
