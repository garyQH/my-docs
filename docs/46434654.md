# i8n的演进

记录一下这两年在前端国际化（`i18n`）上的应用和演进。

在企业级项目中，`i18n` 早已脱离了单纯的 **文本替换** 范畴。

如今的方案，核心痛点在于 **研发体验**（自动提取 / 无感接入）、**加载性能**（按需加载 / `CDN` 分发）以及 **协同效率**（与翻译平台的打通）。

以下结合`Vue`技术栈进行简单的演示。

---

## 1.本地化（静态资源映射）

这是最早先的方案，也是大多数中小型项目的标配。

通过维护多个语言的 `JSON`/`JS` 文件，在运行时通过全局的 `Context` 或特定的 `Hook` 进行响应式替换。

- **核心思想：** 基于 `Key-Value` 的映射。
- **主流工具：** `vue-i18n`、`react-i18next`、`formatjs`。

**示例 Demo**（`Vue 3` + `vue-i18n`）：

```js
// 1. 定义语言包 (locales/zh.ts)
export default {
  nav: {
    home: '首页',
    profile: '个人中心'
  },
  welcome: '欢迎回来, {name}!'
}
```

```js
// 2. 初始化 i18n (main.ts)
import { createI18n } from 'vue-i18n'
import zh from './locales/zh'
import en from './locales/en'

const i18n = createI18n({
  legacy: false, // 启用 Composition API
  locale: 'zh',
  fallbackLocale: 'en',
  messages: { zh, en }
})
```

```js
// 3. 在组件中使用 (App.vue) 
<script setup lang="ts">
import { useI18n } from 'vue-i18n'
const { t, locale } = useI18n()

const switchLang = () => {
  locale.value = locale.value === 'zh' ? 'en' : 'zh'
}
</script>

<template>
  <nav>{{ t('nav.home') }}</nav>
  <p>{{ t('welcome', { name: 'Admin' }) }}</p>
  <button @click="switchLang">Switch Language</button>
</template>
```

**评估：**

- **优点：** 接入简单，生态成熟，支持插槽、复数、日期格式化等基础功能。
- **缺点：** 语言包打包在主构建产物中，随着业务增长，首屏加载体积会迅速膨胀；开发体验差，需要频繁在业务代码和 `JSON` 文件之间横跳，手动维护 `Key`。

---

## 2.工程化（按需加载与自动提取）

为了解决上面的性能和开发体验问题，后面我们在构建层（`Webpack`/`Vite`）引入自动化方案。

### 1. 语言包动态按需加载

不再将所有语言包打包在一起，而是根据用户的当前语言环境，使用 `import()` 动态拉取对应的 `Chunk`。

**示例 Demo：**

```js
// 动态加载语言包的逻辑
const loadedLanguages: string[] = []

export async function loadLanguageAsync(i18n: any, lang: string) {
  // 如果已经加载过，直接切换
  if (loadedLanguages.includes(lang)) {
    i18n.global.locale.value = lang
    return setI18nLanguage(i18n, lang)
  }

  // 动态引入对应语言的 chunk
  const messages = await import(`./locales/${lang}.json`)
  i18n.global.setLocaleMessage(lang, messages.default)
  loadedLanguages.push(lang)
  return setI18nLanguage(i18n, lang)
}
```

### 2. `AST` 代码自动提取与注入

开发人员只需在代码中直接写中文，通过 `Babel` 插件或 `Vite`/`Webpack` 插件，在 `AST` 解析阶段自动提取中文字符串，生成 `Hash` 化的 `Key`，并自动生成更新基础的 `JSON` 语言包。

- **工作流：** 开发者写 `<p>你好</p>` → 编译时插件提取**你好**生成 `Hash`（如 `key_a1b2`）→ 自动替换代码为 `<p>{{ t('key_a1b2') }}</p>` → 将映射关系写入中控 `JSON`。

**评估：**

- **优点：** 首屏体积大幅减小；开发体验极佳，实现了真正的 **代码即文案，对开发者透明**。
- **缺点：** 依赖构建工具的深度定制，`AST` 转换在大型项目中可能会增加一定的打包耗时（不过 `Vite` 体系下利用 `esbuild`/`swc` 可显著缓解）。

## 3.动态下发 + `TMS` 协同 + 边缘计算

在出海业务中，前端代码的发版频率远远跟不上运营修改文案的频率。此时，`i18n` 不再仅仅是前端的工程，而是一个横跨产品、运营、翻译团队的 **全链路系统**。

### 1. `TMS`平台打通

- **流程：** `CI/CD` 流水线在代码 `push` 时，自动扫描提取新增的词条，通过 `API` 推送到公司的 `TMS` 平台。翻译团队或机器翻译（如调用大模型 `API`）在平台上完成多语言翻译。
- **收益：** 研发彻底告别维护多语言 `JSON` 文件的枯燥工作。

### 2. `OTA` 动态下发与 `CDN` 分发

语言包完全从前端代码库中剥离。

前端只保留兜底（`Fallback`）语言。

- **运行时获取：** 前端启动时，向后端的配置中心（如 `Apollo`）或 `CDN` 请求最新的语言包 `JSON`。
- **灰度与热更新：** 运营可以在后台随时修改某句文案，发布后通过 `CDN` 直接生效，无需前端重新构建部署。

**架构示例：**

```js
// 伪代码：结合拉取远程配置的 i18n 方案
async function initApp() {
  const userLang = navigator.language || 'en'
  const version = localStorage.getItem('i18n_version') || '0'

  try {
    // 从 CDN 获取带有 hash 缓存控制的语言包
    const res = await fetch(`https://cdn.company.com/i18n/${userLang}.json?v=${version}`)
    const remoteMessages = await res.json()

    const i18n = createI18n({
      locale: userLang,
      messages: {
        [userLang]: remoteMessages
      }
    })

    app.use(i18n).mount('#app')
  } catch (e) {
    // 降级：使用本地打包的兜底文件
    initWithLocalFallback()
  }
}
```

**评估：**

- **优点：** 极致的灵活性，文案修改与代码发版解耦；多团队协同效率最高。
- **缺点：** 基础设施建设成本极高，需要搭建完善的 `TMS` 系统、`CDN` 容灾以及前端的异常兜底策略。

## 4.底层规范与深水区

高级的前端 `i18n` 还需要处理文化与习惯的差异，这往往是核心架构设计中的难点。

### 1. 原生 `Intl` API 的深度应用

我本人其实是倾向于利用浏览器原生的 `Intl` 对象来处理日期、时间、货币和数字格式化，而不是引入庞大的 `moment.js` 或自己写正则。这能极大降低 `Runtime` 体积。

**示例 Demo：**

```js
// 货币格式化
const price = 123456.789
// 德国地区，使用欧元
console.log(new Intl.NumberFormat('de-DE', { style: 'currency', currency: 'EUR' }).format(price))
// 输出: 123.456,79 € (注意欧洲习惯的点和逗号用法)

// 相对时间格式化
const rtf = new Intl.RelativeTimeFormat('zh-CN', { numeric: 'auto' })
console.log(rtf.format(-1, 'day')) // 输出: 昨天
```

### 2. `RTL`布局支持

对于阿拉伯语、希伯来语等中东市场语言，整个页面的阅读顺序需要反转。

- **方案：** 使用 `CSS` 逻辑属性（`Logical Properties`）。弃用 `margin-left`、`padding-right`，全面改用 `margin-inline-start`、`padding-inline-end`。
- 当 `HTML` 标签加上 `dir="rtl"` 时，浏览器会自动处理镜像翻转。

```css
/* 传统写法 - 无法自适应 RTL */
.box { margin-left: 20px; }

/* 现代写法 - 配合 dir="rtl" 自动翻转 */
.box { margin-inline-start: 20px; }
```

---

## 总结

从架构演进的角度来看：

1. **早期 / 小型项目**：依靠生态插件（`vue-i18n`），手动维护 `JSON`，重点是跑通流程。
2. **成长型 / 中型项目**：依靠工程化（`Vite` 插件 / `AST`），实现按需加载和自动提取，重点是优化开发体验和性能。
3. **大厂 / 出海业务**：依靠平台化（`TMS` + `CDN` 动态下发 + `LLM` 辅助翻译），实现业务与研发解耦，重点是交付效率和动态运营。
4. **技术深水区**：深度融合 `Intl` 原生 `API` 与 `CSS` 逻辑属性，实现文化层面的完美本地化。
