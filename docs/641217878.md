## 一、Socket.IO 如何实现断网重连

原生的 `WebSocket` 协议本身是非常**脆弱**的：一旦网络波动导致连接断开，它的状态就会变成 `CLOSED`，然后就彻底"死"了，不会有任何自我修复的行为。

**`Socket.IO`** 并不是修改了浏览器的底层逻辑，而是在原生 `WebSocket` 之上套了一层极其精密的**管理机制**。

---

### 1. 核心机制：状态机与事件监听

`Socket.IO` 在内部维护了一个复杂的**状态机**。它并不只是简单地调用 `new WebSocket()`，而是把这个过程包装在一个名为 `Manager` 的类中。

* **监听 `onclose`:** 当原生的 `WebSocket` 触发 `onclose` 事件时，`Socket.IO` 的内部监听器会立刻捕捉到。
* **启动计时器:** 一旦检测到连接断开，它不会立刻重连（防止瞬间产生大量无效请求），而是启动一个计时器。

---

### 2. 指数退避算法 (`Exponential Backoff`)

如果网络真的断了，立刻重连大概率还是会失败。`Socket.IO` 使用了一种聪明的策略叫**指数退避**：

1.  **第一次尝试:** 1 秒后重连。
2.  **第二次尝试:** 2 秒后。
3.  **第三次尝试:** 4 秒后。
4.  **以此类推:** 间隔时间会越来越长，直到达到一个上限（默认 5 秒）。

> **为什么要这样做？** > 想象一下，如果大面积断网，成千上万个客户端都在每秒重连 10 次，那么当网络恢复的一瞬间，服务器会被这股"流量洪水"直接冲垮（这叫 **惊群效应**）。

---

### 3. "心跳"检测 (`Heartbeat` / `Ping-Pong`)

有时候网络并没有完全断开，只是由于路由器丢包或者运营商干扰，导致连接"假死"（僵尸连接）。

原生 `WebSocket` 可能反应不过来。

`Socket.IO` 引入了 **`Ping`/`Pong` 机制**：

* **服务器说:** "嘿，你在吗？" (`Ping`)
* **客户端回:** "我在呢！" (`Pong`)
* 如果客户端在规定时间内（比如 25 秒）没收到服务器的 `Ping`，或者服务器没收到 `Pong`，`Socket.IO` 就会**主动断开**当前的烂连接，然后触发重连流程。

---

### 4. "黑科技"：`Buffer` 机制（消息不丢失）

自动重连最牛的地方在于：**在你重连的那几秒钟里，你发送的消息并不会丢失。**

* 当你调用 `socket.emit()` 时，如果此时处于断开状态，`Socket.IO` 会把这些消息暂存到一个**离线队列 (`Packet Buffer`)** 中。
* 一旦连接重新建立成功，它会自动把队列里的消息按顺序"重发"出去。
* 从开发者的角度看，你只是发了个消息，完全没感知到中间断过网。

---

### 5. 源码级逻辑伪代码

如果我们自己用原生 `JS` 仿写一个简易版的"自动重连"，大概长这样：

```js
function connectWithRetry(url) {
    let ws = new WebSocket(url);

    ws.onclose = () => {
        console.log("连接断了，准备重连...");
        // 关键：在这里设置一个定时器，重新调用自己
        setTimeout(() => {
            connectWithRetry(url); 
        }, 5000); // 5秒后重试
    };

    ws.onerror = (err) => {
        ws.close(); // 报错也触发关闭逻辑，统一去重连
    };

    return ws;
}
```

**`Socket.IO` 的伟大之处就在于它把这套逻辑写得极其健壮，处理了各种边界情况（比如浏览器休眠、网络切换、服务器压力缓解等）。**

---

### 总结

`Socket.IO` 是原生的**高级包装**。

它通过监听底层事件、使用定时器重试、配合心跳包检测以及离线缓存，把原本 **一次性** 的连接变成了一个 **坚不可摧** 的通道。

## 二、Socket.IO 是如何判断“网络已经恢复”的?

`Socket.IO` 并不是通过某种"魔法"直接感知到你的网线插上了，它其实是一套**主动侦测 + 多层兜底**的组合拳。

我们可以把它的判断逻辑拆解为以下四个层面：


### 1. 浏览器的"告密"：网络事件监听

最基础的一层是利用浏览器原生的能力。`Socket.IO` 内部（或者说它依赖的 `engine.io`）会监听浏览器提供的网络状态事件：

* **`online` 事件**: 当浏览器检测到网络接口恢复（比如 `Wi-Fi` 连上了），会触发这个事件。
* **`offline` 事件**: 掉线时触发。

当 `online` 触发时，`Socket.IO` 会立刻意识到："嘿，可能有机会了！"它会打破当前的等待计时器，**立即尝试一次重连**，而不是傻傻地等到下一个指数退避的时间点。


### 2. 核心招式：心跳包机制 (`Heartbeat`)

这是判断"网络是否真正通畅"最可靠的方法。`Socket.IO` 在连接建立后，会不断地进行 **`Ping`/`Pong`** 交换。

* **发送 `Ping`**: 客户端定期向服务器发送一个极小的包（默认每 25 秒）。
* **期待 `Pong`**: 服务器收到 `Ping` 后必须立刻回一个 `Pong`。
* **判断恢复**:
    * 如果连接断了，`Socket.IO` 进入"重连模式"。
    * 在重连模式下，它会不断尝试建立新的连接。
    * **只有当一个新的 `WebSocket` 握手成功，且成功收到第一个来自服务器的响应包时**，`Socket.IO` 才会正式宣布："网络已恢复（`Reconnected`）！"


### 3. 这里的"坑"：`TCP` 的半打开状态

有时候你的 `Wi-Fi` 信号满格，但其实根本上不了网（比如欠费了，或者路由器死机了）。这时候浏览器可能还觉得它是 `online` 的。

为了对付这种"僵尸连接"，`Socket.IO` 使用了**超时机制 (`Connect Timeout`)**：

1.  它发起一个连接请求。
2.  设置一个定时器（比如 10 秒）。
3.  如果 10 秒内没有完成握手，它会**强行掐断**这次尝试，判定为"网络未恢复"，并等待下一次重试。


### 4. 传输方式的"降级与升级" (`Poling` to `WebSocket`)

这是 `Socket.IO` 的独门绝技。当它尝试恢复连接时，为了保证成功率，它不一定非要死磕 `WebSocket`。

1.  **先用 `HTTP` 长轮询 (`HTTP Long Polling`)**：因为它最稳定，不容易被防火墙拦截。如果 `HTTP` 能通，说明"网络基础层"恢复了。
2.  **再升级到 `WebSocket`**：在 `HTTP` 通了之后，它会在后台默默测试 `WebSocket`。如果测试成功，再无缝切换过去。

> **所以，它判断"恢复"的标准不是"网线接好了"，而是"数据能跑通了"。**


### 5. 开发者如何感知这个过程？

`Socket.IO` 把这些复杂的内部判断转换成了简单的**事件**。你可以在代码里监听这些"恢复瞬间"：

```js
// 监听重连尝试
socket.io.on("reconnect_attempt", (attempt) => {
  console.log(`正在尝试第 ${attempt} 次重连...`);
});

// 监听重连成功 —— 这就是 Socket.IO 确认"网络恢复"的时刻
socket.io.on("reconnect", (attempt) => {
  console.log(`太棒了！历经 ${attempt} 次重试，网络终于恢复了！`);
});

// 监听重连失败（超过最大重试次数）
socket.io.on("reconnect_error", (error) => {
  console.log("重连还是失败了，可能服务器挂了");
});
```

### 总结

`Socket.IO` 判断网络恢复的逻辑是：**"浏览器说有网了" + "握手协议跑通了" + "心跳包对上了"**。

只有这三者（特别是后两者）满足，它才会放心地告诉你连接已恢复。
