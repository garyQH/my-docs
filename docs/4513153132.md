## 前言

在 **Webpack 5** 之前，构建缓存主要依赖第三方插件（如 `cache-loader`）或手动配置 `hard-source-webpack-plugin`。

**Webpack 5** 正式引入了内置的持久化缓存（`Persistent Caching`），这让二次构建（`Rebuild`）的速度提升了 **5x** 到 **10x**，在大厂的巨型项目（千万级行代码）中效果尤为震撼。

以下是它缩短时间的底层逻辑：

## 一、 从“全量内存”到“磁盘复用”

在没有持久化缓存时，`Webpack` 每次启动都要经历完整的生命周期：**读取文件** -> **语法解析 (AST)** -> **转换 (Loader)** -> **依赖分析** -> **代码压缩**。

这些过程极其消耗 `CPU`。

持久化缓存的介入点：

### 第一步：序列化存储。

`Webpack` 会将 `Module Graph`（模块图谱）和已经处理好的代码块（`Chunks`）序列化为**二进制文件**，存放在磁盘的 `.cache/webpack` 目录下。

### 第二步：指纹校验。

二次构建启动时，`Webpack` 不会直接开跑 `Loader`，而是先对比本地文件的 **内容哈希（`Content Hash`）**。

### 第三步：跳过计算。

如果文件没变，`Webpack` 直接从 **磁盘** 读取之前的编译结果，跳过最耗时的 `Babel` 转换和 `Terser` 压缩阶段。

## 二、 核心机制：模块图谱（Module Graph）的恢复

`Webpack` 最复杂的部分不是处理单个文件，而是理清几千个文件之间的** 引用关系**。

- 旧版本： 即使只改了一个文件，`Webpack` 也要重新爬取整个依赖树来确保路径正确。

- 新版本： 持久化缓存记录了整个 `Module Graph` `的快照。Webpack` 只需要识别出受影响的一小个分支，剩下的“树干”直接从缓存中恢复。这种增量构建的思想是速度飞跃的关键。

### 三、 缓存的“多层”结构

`Webpack` 的缓存不只是存 `JS` 代码，它分为多个颗粒度：

| 缓存层级      | 缓存内容 |   缩短的时间点 |
| ----------- | ----------- |---------|
| **Resolving**     | 模块路径解析结果（如 `import '@utils'` 对应哪个物理路径）       |  减少磁盘 IO 查找时间 | 
| **Building**   | `Loader` 转换后的结果（`AST`、`Source Map`）        | 最核心： 跳过 `Babel/TS` 编译耗时 |
| **Providing**   | 模块导出的变量信息       |  加速依赖链接（`Linking`）过程 |
| **Optimization**   | `Minify`（压缩）和模块合并的结果      |  跳过极慢的 `Uglify/Terser` 过程

### 四、如何正确开启与配置

在你的 `webpack.config.js` 中，只需要极简的配置：

```js
module.exports = {
  cache: {
    // 1. 设置为磁盘存储
    type: 'filesystem', 
    
    // 2. 只有当这些配置文件变动时，缓存才失效
    buildDependencies: {
      config: [__filename], 
    },
    
    // 3. 区分环境，避免开发和生产环境缓存混用
    name: process.env.NODE_ENV === 'production' ? 'prod-cache' : 'dev-cache',
  },
};
```

### 五、 为什么有时候缓存会“失效”？

`Webpack` 拥有一套非常严谨的失效检测机制。

以下变动会触发缓存失效（冷启动）：

- `Webpack` 或 `Loader/Plugin` 版本升级。
- 配置参数修改（如改了 `publicPath`）。
- `Node.js` 版本更迭。
- 命令行参数变化。

### 总结：大厂如何压榨这个特性？

在大厂的 `CI/CD` 流程中，通常会将 `.cache/webpack` 文件夹打包上传到远端存储（如 `S3` 或 `OSS`）。

当下一位开发者拉取代码或流水线再次启动时，先下载这个缓存包。

这样即便是不同机器的首次构建，也能享受到二次构建的“秒开”体验。
