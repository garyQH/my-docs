# qiankun

了解 `qiankun` 的实现，本质上是理解它如何在上层封装了 `single-spa`，并解决了微前端最棘手的三个问题：

- **资源加载**（`HTML Entry`）
- **运行沙箱**（`JS Sandbox`）
- **样式隔离**（`CSS Isolation`）

我们可以把 `qiankun` 看作是一个增强版的浏览器渲染引擎，它在主应用中模拟了一套子应用的运行环境。

---

## 1. 资源加载机制：`HTML Entry`

这是 `qiankun` 相比 `single-spa` 最大的改进。

`single-spa` 要求手动指定子应用的 `JS` 资源地址，而 `qiankun` 只需要你提供子应用的 `URL`。

- **解析过程**：它内部使用了 `import-html-entry` 库。通过 `fetch` 获取子应用的 `HTML` 内容。
- **提取关键要素**：利用正则表达式解析出 `HTML` 中的 `script`（`JS` 脚本）和 `link`（`CSS` 样式）。
- **动态执行**：将提取出的 `CSS` 插入到主应用的 `<head>` 中，将 `JS` 内容通过 `eval` 或 `new Function` 的方式在沙箱环境中执行。

**优点**：像使用 `<iframe>` 一样简单，子应用发布后，主应用不需要修改任何配置就能自动获取最新的资源。

---

## 2. 核心隔离：`JS` 沙箱（`JS Sandbox`）

为了防止子应用 `A` 污染全局变量（如 `window.a = 1`）影响到子应用 `B` 或主应用，`qiankun` 设计了三套沙箱模式：

### ① `SnapshotSandbox`（快照沙箱）

适用于不支持 `Proxy` 的旧版浏览器（`IE11`）。

- **原理**：在子应用激活时，记录当前 `window` 的快照。在子应用销毁时，对比当前状态与快照，将差异部分存起来，并把 `window` 还原。
- **缺点**：性能较差，且不支持同时运行多个子应用。

### ② `LegacySandbox`（单实例代理沙箱）

这是 `qiankun` 早期使用的 `Proxy` 方案。

- **原理**：使用 `Proxy` 监听对 `window` 的修改，并记录在 `addedPropsMap` 和 `modifiedPropsMap` 中。
- **特点**：虽然快，但它依然是直接操作原生 `window` 对象，所以同一时间只能激活一个子应用。

### ③ `ProxySandbox`（多实例代理沙箱）

**这是目前最先进的默认方案。**

- **原理**：它为每个子应用创建一个**虚假的 window 对象**（`Fake Window`）。
- **运作机制**：子应用在读取变量时，先去 `Fake Window` 找，找不到再去真实的 `window` 找；但在**写入**变量时，只会写在自己的 `Fake Window` 里。
- **优点**：真正的隔离，支持同一个页面同时运行多个子应用。

---

## 3. 样式隔离：`CSS Isolation`

样式污染是微前端最头疼的问题。`qiankun` 提供了两种官方方案：

- `Strict Web Components Setting`：

  利用浏览器原生的 `Shadow DOM`。它会将子应用包裹在一个 `Shadow Root` 中，样式在物理层面被完全阻断。

  > **痛点**：兼容性问题，且某些 `UI` 框架（如 `Ant Design`）的弹窗会跳出 `Shadow DOM` 导致样式丢失。

- `Experimental Scoped CSS`：

  运行时动态改写 `CSS` 选择器。类似于 `Vue` 的 `scoped`，给子应用的所有样式加上前缀。

  > **示例**：`.foo { color: red; }` 变成 `div[data-qiankun="app1"] .foo { color: red; }`。

---

## 4. 生命周期管理

`qiankun` 严格遵循 `single-spa` 的协议，但在加载过程中增加了更多的钩子：

1. `bootstrap`：子应用代码首次加载，进行初始化（只触发一次）。
2. `mount`：子应用挂载到主应用的容器 `DOM` 中。此时会触发 `render`。
3. `unmount`：子应用卸载。此时会清理沙箱、移除样式、销毁组件实例。
4. `update`（可选）：用于主应用向子应用传参。

---

## 5. 通信机制：`Actions` 通信

`qiankun` 官方提供了一个简单的发布/订阅模型：`initGlobalState`。

- 主应用定义全局状态，并下发 `onGlobalStateChange`（监听）和 `setGlobalState`（修改）两个方法。
- 子应用在 `mount` 时接收到这些方法。
- **数据流**：任何一方修改状态，其他方都能实时感知。

---

## 总结：为什么选 `qiankun`？

`qiankun` 的精髓在于它做到了**无感接入**。它通过 `HTML Entry` 解决了配置繁琐的问题，通过 `Proxy` 沙箱解决了安全性问题，通过生命周期钩子解决了协同问题。
